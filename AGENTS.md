# AGENTS.md

## 1. 문서 목적

이 문서는 이 저장소 및 그 하위 프로젝트들에 공통 적용되는 **최상위 작업 기준 문서(공통 헌법)**이다.

이 문서는 다음 목적을 가진다.

- 문서 작업과 구현 작업의 공통 운영 규칙을 정한다.
- 개발자와 AI의 역할과 책임을 분리한다.
- 문서 우선 원칙과 구조 우선 원칙을 유지한다.
- 현재 단계에 필요한 최소 수준의 작업만 진행하도록 범위를 통제한다.
- 변경 관리와 상태 반영 원칙을 고정한다.
- 반복 검증을 통해 다른 프로젝트에도 적용 가능한 운영 템플릿의 기준으로 활용한다.

이 문서는 상세 설계 문서나 현재 상태 문서를 대체하지 않는다.  
이 문서는 프로젝트 전반의 **작업 규칙 / 협업 운영 기준 / AI 운영 기준**을 정의한다.

---

## 2. 프로젝트 성격

butgo-specdrive는 개발 스펙 문서를 기반으로 AI 협업을 실행하는 도구다.

현재 이 저장소는 다음 성격으로 이해해야 한다.

- 완성된 SaaS 제품 저장소가 아니다.
- 현재는 **core workflow / engine / repo-local Codex skill 중심 운영체계 검증 단계**다.
- 실제 애플리케이션 프로젝트를 함께 두고 specdrive의 운영 방식을 검증하는 통합 작업 저장소다.
- 현재 AI 엔진 기준은 **Codex**다.
- 현재 실행 기준은 **repo-local Codex skill 직접 사용**이다.

이 저장소를 일반적인 웹 서비스, 백오피스 SaaS, 협업 플랫폼 UI 프로젝트처럼 해석하지 말아야 한다.

- 루트 수준의 공통 문서

예:

- `docs/specdrive/AGENTS.md`
- `docs/projects/board/AGENTS.md`

## 5. 우선순위 규칙

문서 간 규칙이 충돌할 경우 우선순위는 다음과 같다.

1. 루트 `AGENTS.md`
2. 해당 영역의 전용 `AGENTS.md`
3. 개별 문서
4. 작업 메모, 임시 노트, 대화 초안

즉, 하위 문서는 상위 규칙을 위반할 수 없다.  
하위 문서는 상위 규칙을 구체화한 내용이어야 하며, 상위 규칙과 충돌해서는 안 된다.

## 6. 이 저장소를 보는 기본 관점

이 저장소는 크게 다음 축으로 이해한다.

### 6.1 specdrive
specdrive는 **어떻게 협업할 것인가**를 다루는 도구 영역이다.

주요 관심사:

- 문서 기반 AI 협업 흐름
- 세션 시작 규칙
- 문맥 복구
- 프롬프트 실행 흐름
- 문서 검토와 동기화
- repo-local Codex skill 구조
- history / phase / cycle 관리
- 반복 가능한 작업 절차 정규화

### 6.2 projects
projects는 **무엇을 만들 것인가**를 다루는 실제 애플리케이션 작업공간이다.

여기에는 다음이 포함될 수 있다.

- 요구사항
- 설계
- 구현 계획
- 상태
- 히스토리
- 애플리케이션별 결정 사항
- 검증용 실제 프로그램 또는 실제 프로젝트 문서

### 6.3 standards
standards는 projects 하위에서 공통으로 사용하는 개발 기준 문서군이다.

예:

- naming
- java coding
- db
- api
- package structure
- git policy

## 7. 핵심 작업 방향

이 저장소의 작업 방식은 **문서 우선의 AI 협업 개발**을 기본 전제로 한다.

핵심 흐름은 다음과 같다.

- Spec
- Prompt
- Code
- History

즉, 코드를 바로 늘리는 방식보다 다음 흐름을 우선한다.

1. 먼저 문서로 구조와 경계를 정한다.
2. 그 문서를 기준으로 AI 입력을 구성한다.
3. 그 결과를 코드나 후속 문서에 반영한다.
4. 판단과 변경 이력을 남긴다.

## 8. 작업 기본 원칙

프로젝트는 다음 원칙을 따른다.

- 문서가 기준이다.
- 구현은 문서를 따른다.
- 문서 간 역할은 겹치지 않게 유지한다.
- 현재 단계에 필요한 최소 수준부터 시작한다.
- AI는 보조 수단이며, 최종 판단과 반영 책임은 개발자에게 있다.
- 현재 범위와 다음 전환 시점의 통제권은 개발자에게 있다.
- 자동 진행보다 개발자의 중간 점검과 수정 가능성을 우선한다.
- 구조는 단기 편의보다 장기 확장 가능성을 우선한다.
- 아직 결정되지 않은 내용은 확정처럼 쓰지 않는다.
- 현재 범위를 넘는 미래 확장을 먼저 구현하지 않는다.
- 현재 상태와 구현 상태는 상태 문서에서 확인하고, 이 문서에는 운영 기준만 유지한다.
- 이 저장소는 현재 프로젝트 구현뿐 아니라 반복 실행 가능한 운영 템플릿 검증용 기준 저장소로도 사용한다.
- 초기 상태로 되돌리는 작업은 개발자가 직접 요청할 때만 수행한다.
- AI는 신규 문서 생성, 문서 역할 변경, 작업 단계 전환, 요구사항에서 설계 또는 설계에서 구현 계획으로 넘어가는 전환점에서는 먼저 개발자에게 진행 여부를 확인한다.
- 단, 개발자가 해당 전환이나 신규 문서 작성을 명시적으로 요청한 경우에는 그 요청 범위 안에서 진행할 수 있다.

### 코드/주석 원칙
- 생성 코드의 주석은 자명한 코드 설명보다 복잡한 분기, 경계 조건, 구조 의도를 이해하기 위한 설명을 우선한다.
- 생성 코드의 주석은 특별한 이유가 없으면 한글로 작성한다.
- 이 주석 원칙은 운영 코드뿐 아니라 테스트 파일에도 동일하게 적용한다.
- 필요하면 파일 상단에 해당 파일의 역할, 책임 경계, 문서 기준과의 연결점을 짧게 설명하는 주석을 둘 수 있다.
- 필요하면 함수, 메서드, 주요 타입 선언 앞에 입력/출력 의미와 핵심 의도를 이해할 수 있는 설명 주석을 둘 수 있다.
- 여러 단계의 판단이나 예외 처리가 섞인 구간에는 왜 이런 순서와 조건을 쓰는지 이해할 수 있도록 블록 단위 설명 주석을 둘 수 있다.
- 주석은 유지보수에 실제 도움이 되는 수준으로만 추가하고, 코드만 읽어도 자명한 내용은 반복 설명하지 않는다.

## 9. 개발자와 AI의 역할 원칙

### 9.1 개발자
개발자는 다음 책임을 가진다.

- 작업 범위와 우선순위를 결정한다.
- 현재 단계와 다음 전환 시점을 판단한다.
- 문서와 구현의 최종 승인 책임을 가진다.
- 구조 경계와 금지 규칙의 유지 여부를 판단한다.
- AI 결과를 그대로 채택할지, 수정할지, 보류할지 결정한다.

### 9.2 AI
AI는 다음 역할을 수행한다.

- 문서 초안 보강
- 문서 검토 및 정리
- 구조/중복/충돌 점검
- 구현 보조
- 테스트/정리/설명 보조

AI는 보조 수단이며, 다음을 해서는 안 된다.

- 명시되지 않은 결정을 확정처럼 다루는 것
- 현재 범위를 넘는 내용을 먼저 진행하는 것
- 개발자 확인 없이 신규 문서를 만들거나 문서 단계 전환을 확정하는 것
- 기존 문서의 역할을 임의로 합치거나 삭제하는 것
- 구조 경계를 무너뜨리는 방향으로 단순 편의 구현을 밀어붙이는 것

## 10. 문서 작성 원칙

### 10.1 문서마다 역할이 분명해야 한다
문서는 다음 중 하나 이상의 분명한 역할을 가져야 한다.

- 방향 제시
- 구조 정의
- 규칙 정의
- 절차 정의
- 상태 기록
- 변경 이력 관리

역할이 모호한 문서는 분리 또는 정리 대상이다.

### 10.2 문서는 현재 결정을 먼저 담아야 한다
가능성 나열보다 현재 채택한 결정을 우선 기록한다.

### 10.3 현재 범위와 후속 범위를 분리한다
다음을 혼합하지 않는다.

- 현재 확정
- 보류
- 후속 후보
- 장기 아이디어

### 10.4 추상어만으로 끝내지 않는다
“유연하게”, “확장 가능하게”, “체계적으로” 같은 표현만으로 마무리하지 않고  
실제 적용 규칙 또는 경계로 연결한다.

### 10.5 AI가 읽기 쉬운 구조를 유지한다
문서는 사람뿐 아니라 AI 입력으로 사용되므로 다음을 유지한다.

- 제목 체계 일관성
- 용어 일관성
- 중복 최소화
- 참조 문서 명시
- 현재 단계와 후속 단계를 구분

### 10.6 Markdown 코드블록 중첩을 피한다
Markdown 문서, preview, copy prompt 를 작성할 때는 fenced code block 중첩으로 화면이 깨지지 않게 한다.

기준은 다음과 같다.

- 문서 preview 전체를 감쌀 때는 triple backtick 대신 `~~~markdown` 을 우선 사용한다.
- preview 내부의 짧은 예시는 가능한 한 inline code 또는 들여쓰기 코드블록으로 표현한다.
- preview 내부에 JSON, text, shell 예시처럼 fenced code block 이 필요하면, 바깥 preview fence 와 다른 문자를 사용한다.
- copy-ready prompt 안에 전체 문서 preview 를 다시 넣지 않는다.
- note template 처럼 prompt 내부에 Markdown 예시가 필요하면 `~~~markdown` 을 사용해 바깥 triple backtick 과 충돌하지 않게 한다.
- 하나의 응답 안에서 preview, approval prompt, completion prompt 는 가능한 한 별도 블록으로 분리한다.

## 13. 작업 단계 구분

이 저장소의 작업 흐름은 핵심 작업 단계로 `doc`, `dev` 를 두고,  
세션 운영 단계 `session`, 전달 단위 생성 단계 `git` 을 별도로 둔다.

### 13.1 doc 단계
문서를 구현 가능한 상태로 만드는 단계다.

핵심 흐름:

- reinforce
- confirm
- history-save

기본 순서:

1. 개발자 초안 작성
2. AI 문서 보강
3. 개발자 검토
4. 문서 확정
5. history 저장

### 13.2 dev 단계
확정된 문서를 실제 개발 작업 단위로 실행하는 단계다.

핵심 흐름:

- task-split
- phase
- cycle
- status

기본 순서:

1. 확정된 문서를 기준으로 task 분해
2. 현재 phase 설정
3. 현재 cycle 설정
4. task 진행
5. 상태 갱신 및 검토

### 13.3 session 단계
세션 시작/종료와 작업 전달을 돕는 운영 단계다.

핵심 흐름 예:

- session start-lite
- session start
- session status
- session save

기본 순서 예:

1. 세션 시작 시 현재 상태 복구
2. 필요한 경우 `doc` / `dev` 작업 수행
3. 세션 종료 시 현재 변경과 다음 진입점 정리
4. 필요 시 Git/PR 메시지 초안 생성

원칙:

- `session` 은 메타 운영 단계다.
- `session` 은 문서 확정이나 개발 실행 자체를 대신하지 않는다.
- `session save` 는 `doc history-save` 를 대체하지 않는다.

### 13.4 git 단계
Git commit과 GitHub PR 같은 전달 단위 초안을 만드는 단계다.

핵심 흐름 예:

- git-commit
- github-pr

원칙:

- `git` 단계는 Git 전달 단위 생성과 승인 기반 실행에 집중한다.
- `git-commit` 은 현재 브랜치와 변경 파일을 확인하고 commit/push 제안을 준비한다.
- `github-pr` 은 push된 브랜치를 기준으로 PR 초안을 만들고, 명시 승인 후에만 PR을 생성한다.

### 13.5 단계 혼합 금지
문서 단계와 개발 단계를 섞어 다루지 않는다.

- 문서가 확정되지 않았으면 먼저 `doc` 단계에서 다룬다.
- `dev` 단계는 confirm 된 문서를 기준으로 시작한다.
- 문서 보강 문제를 개발 실행 문제처럼 다루지 않는다.
- 개발 상태 문제를 문서 정의 문제처럼 섞지 않는다.
- `session` 단계는 세션 복구/저장/전달 보조로 유지하고, `doc` / `dev` 내부 로직을 대신하지 않는다.
- `git` 단계는 Git 전달 단위 생성으로 유지하고, `session` / `doc` / `dev` 내부 로직을 대신하지 않는다.
- AI는 `doc` 내부에서도 요구사항 작성, 설계 작성, 구현 계획 작성처럼 문서 역할이 바뀌는 지점에서는 개발자 확인을 먼저 받는다.
- AI는 기존 문서 보강과 신규 문서 생성을 구분하고, 신규 문서 생성은 개발자의 명시 요청 또는 확인 이후에 진행한다.

---

## 14. AI 협업 원칙

### 14.1 AI는 재설계자가 아니라 보강자/검토자다
명시적으로 요청되지 않는 한,  
AI는 문서를 처음부터 다시 설계하기보다 현재 문서를 기준으로 보강·정리·검토한다.

### 14.2 최소 변화 원칙
AI는 기존 구조를 유지할 수 있으면 유지한다.

### 14.3 문서 역할을 넘지 않는다
AI는 공통 문서를 검토하다가 특정 프로젝트 설계로 과도하게 확장하지 않는다.  
반대로 프로젝트 문서를 검토하다가 전체 체계를 재설계하지도 않는다.

### 14.4 결정과 아이디어를 구분한다
현재 채택한 내용과 제안 수준의 아이디어를 섞지 않는다.

### 14.5 문서 묶음 검토를 활용한다
개별 문서 품질과 문서 간 정합성은 다르므로 필요 시 나눠서 검토한다.

- 단건 검토: 문서 자체 품질 보강
- 묶음 검토: 문서 간 역할/용어/경계 정합성 확인

### 14.6 AI 결과는 바로 정답이 아니다
AI 산출물은 초안, 보강안, 리뷰안으로 취급하며  
개발자가 최종 승인한다.

---

## 15. 구현 원칙

### 15.1 구현은 문서를 따른다
문서가 기준이고, 구현은 그 기준을 반영한다.

### 15.2 코드 구조는 문서 구조를 침범하지 않는다
프레임워크, DB, UI, 저장 방식 때문에  
문서 체계와 책임 분리가 왜곡되면 안 된다.

### 15.3 현재 단계에 맞는 단순함을 우선한다
나중에 필요할 수 있다는 이유만으로  
복잡한 추상화, 분산 구조, 다중 런타임 설계를 조기 도입하지 않는다.

### 15.4 전환 가능성은 열어두되, 현재 구현은 명확하게 한다
미래 전환 가능성을 고려할 수는 있지만,  
그 가능성 때문에 현재 구조를 애매하게 만들지 않는다.

---

## 16. 변경 관리 원칙

### 16.1 중요한 변경은 흔적을 남긴다
구조, 역할, 정책, 흐름에 영향을 주는 변경은  
가능한 한 문서 히스토리나 변경 기록을 남긴다.

### 16.2 변경 영향 범위를 확인한다
하나의 문서를 수정할 때는 다음을 확인한다.

- 어떤 상위 규칙과 연결되는가
- 어떤 하위 문서가 영향을 받는가
- 용어 충돌이 생기는가
- 현재 단계 정의가 바뀌는가

### 16.3 구조 변경과 상태 변경을 구분한다
- 구조 변경: 문서 역할, 책임, 분류, 체계 변경
- 상태 변경: 현재 단계, 진행도, 적용 여부 변경

이 둘은 같은 수정처럼 보여도 의미가 다르므로 구분해서 다룬다.

### 16.4 변경 전에 위치를 먼저 판단한다
내용이 맞는지 이전에,  
그 내용이 **어느 문서에 있어야 하는지**부터 판단한다.

---

## 17. 상태 일치 원칙

현재 단계 분류와 활성 상태를 담는 문서들은 가능한 한 서로 일치해야 한다.

즉, 현재 작업 기준이 바뀌면 관련 상태 문서도 함께 갱신한다.

다만 이 문서는 상태 문서가 아니므로,  
구체적인 현재 상태를 직접 기록하기보다 상태 문서의 존재와 역할을 유지하는 기준으로 사용한다.

---

## 18. 히스토리 원칙

### 18.1 히스토리는 저장 가치가 있는 판단을 남긴다
모든 사소한 수정까지 다 저장할 필요는 없지만,  
다음은 남길 가치가 높다.

- 구조 변경
- 규칙 변경
- 문서 역할 조정
- 단계 전환
- 중요한 설계 판단
- AI 협업 절차 변경

### 18.2 결과뿐 아니라 판단 흐름도 중요하다
최종본만큼 중요한 경우에는  
왜 그렇게 바뀌었는지도 남긴다.

### 18.3 히스토리는 현재 문서를 대체하지 않는다
히스토리는 참고와 추적용이지,  
현재 기준 문서 자체를 대신하지 않는다.

### 18.4 `docs/history/**` 는 기본 관리 대상이 아니다
`docs/history/**` 는 현재 기준 문서가 아니라 과거 판단과 산출물을 보존하는 이력 영역으로 본다.

따라서 다음을 기본 원칙으로 둔다.

- `docs/history/**` 는 평소 세션 복구나 문서 정비의 기본 점검 범위에 넣지 않는다.
- 현재 규칙, 현재 상태, 현재 문서 구조를 정리할 때는 먼저 현재 기준 문서와 실행 자산을 본다.
- `docs/history/**` 는 개발자가 특정 이력 확인, 근거 추적, 비교 검토를 명시적으로 요청한 경우에만 확인한다.
- 과거 명령명, 과거 경로, 과거 산출물 명칭은 당시 이력 보존을 위해 유지할 수 있다.

## 19. 프롬프트/컨텍스트 원칙

### 19.1 AI에게는 필요한 문맥만 준다
무조건 많은 문서를 주입하기보다  
현재 작업에 필요한 기준 문서와 관련 문서를 묶어 제공한다.

### 19.2 공통 기준을 먼저 준다
가능하면 다음 순서로 맥락을 준다.

1. 루트 `AGENTS.md`
2. 해당 영역 전용 `AGENTS.md`
3. 대상 문서
4. 관련 참조 문서

### 19.3 프롬프트는 작업 종류에 맞게 나눈다
다음을 구분한다.

- 초안 작성용
- 보강용
- 검토용
- 정합성 확인용
- 후처리 확인용

### 19.4 프롬프트도 운영 자산이다
프롬프트는 일회성 문장이 아니라  
반복 가능한 협업 자산으로 관리한다.

## 20. 검토 원칙

### 20.1 검토는 재창조가 아니다
검토의 목적은 현재 문서를 기준으로 모순, 누락, 과설계, 역할 혼합을 찾는 것이다.

### 20.2 좋은 검토는 범위를 넘지 않는다
대상 문서 검토에서 무관한 영역 전체 재설계로 튀지 않는다.

### 20.3 검토는 계층을 본다
문서를 볼 때 항상 다음을 확인한다.

- 이 내용이 공통인가
- 전용인가
- 개별 구현 판단인가

### 20.4 검토 결과는 수정 우선순위를 드러내야 한다
예:

- 반드시 수정
- 수정 권장
- 선택 사항
- 후속 검토

## 21. 금지 사항

다음을 지양하거나 금지한다.

- 공통 규칙과 특정 프로젝트 규칙을 한 문서에 뒤섞는 것
- 아직 검증되지 않은 아이디어를 확정 규칙처럼 쓰는 것
- 미래 확장을 이유로 현재 구조를 과도하게 복잡하게 만드는 것
- AI가 기존 문서를 전면 재작성하도록 쉽게 유도하는 것
- 상위 규칙과 충돌하는 하위 전용 규칙을 두는 것
- 문서 역할이 중복되는데도 정리하지 않고 계속 누적하는 것
- 현재 단계에서 SaaS UI/멀티 엔진/과한 추상화를 먼저 구조에 집어넣는 것

## 22. 권장 작업 흐름

기본 권장 흐름은 다음과 같다.

1. 작업 대상과 범위를 정한다.
2. 관련 상위 문서를 확인한다.
3. 현재 문서를 초안 또는 수정한다.
4. AI로 보강 또는 검토한다.
5. 개발자가 최종 판단한다.
6. 필요 시 연관 문서를 반영한다.
7. 중요한 변경은 히스토리를 남긴다.

AI가 작업을 진행할 때는 다음 확인 절차를 추가로 따른다.

- 현재 문서 내부 보강인지, 신규 문서 생성인지 먼저 구분한다.
- 요구사항 / 설계 / 구현 계획 / 상태 / 히스토리 중 문서 역할이 바뀌면 개발자 확인을 먼저 받는다.
- 세션 시작 후 작업 대상 영역이 정해지면 해당 영역의 `AGENTS.md` 를 추가로 확인한다.
- 확인 없이 진행한 내용은 확정본이 아니라 초안 후보로 취급한다.

## 23. 루트 문서가 다루는 것과 다루지 않는 것

### 이 문서가 다루는 것
- 전체 공통 원칙
- 문서 계층
- AI 협업 공통 규칙
- 변경/상태/히스토리의 기본 원칙
- 문서 역할 분리 기준
- 개발자와 AI의 책임 경계
- doc/dev/session/git 단계 구분의 공통 기준

### 이 문서가 직접 다루지 않는 것
- specdrive 자체의 세부 운영 구조
- board 같은 특정 프로젝트의 상세 아키텍처
- 특정 기능 요구사항
- 특정 프로젝트의 API/DB/패키지 설계
- 특정 시점의 현재 상태 자체
- 개별 명령어의 세부 문법
- Codex exec/skill/자동화의 상세 기술 설계

이런 내용은 반드시 하위 전용 문서 또는 개별 문서에서 다룬다.

## 24. 최종 원칙

이 저장소의 핵심은  
문서를 많이 쌓는 것이 아니라,  
**역할이 분명한 문서와 반복 가능한 AI 협업 흐름을 유지하는 것**이다.

따라서 어떤 작업이든 항상 다음을 먼저 확인한다.

- 이 내용은 공통 규칙인가?
- 이 내용은 전용 규칙인가?
- 이 내용은 실제 구현 판단인가?
- 지금 필요한 최소 구조를 강화하는가?
- 변경 흔적을 남겨야 하는 수준의 수정인가?

이 질문에 답하지 못하면,  
기능 추가보다 문서 위치와 역할부터 다시 정리한다.
