## 1. 소개

butgo-specdrive는 **개발 스펙 문서를 기준으로 AI 협업 개발을 구조적으로 운영하기 위한 도구**다.

이 저장소의 핵심은 단순히 문서를 많이 작성하는 데 있지 않다.  
핵심은 실제 애플리케이션의 개발 스펙 문서를 기반으로,  
세션 복구, 문맥 구성, 프롬프트 실행, 문서 검토, 변경 반영, 작업 분해 같은  
AI 협업 흐름을 반복 가능하게 만드는 데 있다.

즉, specdrive는 프로젝트 내용을 직접 소유하는 시스템이라기보다,  
프로젝트 문서를 읽고 활용하여  
AI 협업 개발을 통제 가능한 흐름으로 운영하는 도구를 지향한다.

현재는 제품과 운영 방식의 검증을 위해  
`specdrive`와 `projects`를 하나의 저장소 안에서 함께 운영한다.

다만 이것은 최종 구조가 아니라,  
실제 적용과 검증을 위한 현재 단계의 운영 방식이다.  
장기적으로는 `specdrive`는 독립된 도구/플랫폼으로,  
`projects`는 specdrive가 다루는 개별 애플리케이션 작업공간으로  
분리 가능한 구조를 지향한다.

## 2. 왜 butgo-specdrive를 만드는가

기존 개발에서는 보통 다음 흐름이 중심이었다.

- DB 설계
- 인터페이스 정의
- 아키텍트가 만든 프레임워크 위에서 구현
- 코드와 개발자 판단을 통해 실제 구조 확정

이 방식은 일정 규모까지는 유효했지만, AI를 본격적으로 활용하는 개발에서는 한계가 있다.

AI는 코드를 빠르게 생성할 수 있지만, 다음이 불분명하면 결과가 쉽게 흔들린다.

- 프로젝트의 목적
- 현재 범위
- 구조적 원칙
- 문서와 코드의 관계
- 작업 순서
- 사람 확인 시점
- 현재 상태와 다음 진입점

즉 AI 시대에는 단순히 “코드를 빨리 만드는 것”보다  
**무엇을 기준으로 AI가 일하게 할 것인가**가 더 중요해진다.

butgo-specdrive는 이런 문제를 줄이기 위해 시작되었다.

- 개발 스펙 문서를 기준으로 설계와 구현 흐름을 유지한다.
- 정책 문서와 상태 문서를 분리한다.
- 현재 상태와 다음 진입점을 빠르게 복구할 수 있게 한다.
- AI를 자율 개발자가 아니라 통제 가능한 실행 보조자로 사용한다.
- 반복 가능한 개발 운영 체계를 도구 형태로 정리한다.

## 3. butgo-specdrive가 지향하는 개발 방식

butgo-specdrive는 AI를 단순 코드 생성 도구로 보지 않는다.

대신 다음과 같이 본다.

- AI는 문서를 읽고
- 문서를 보강하고
- 문서를 기준으로 작업을 분해하고
- 사람의 확인을 거쳐
- 구현과 검토를 반복 수행하는 협업 주체다

즉 핵심은 “AI에게 바로 구현을 시키는 것”이 아니라,  
**문서를 기준으로 AI 협업 흐름을 정규화하는 것**이다.

---

## 4. 우리가 가려는 SDD 방향

butgo-specdrive에서 SDD는 단순히 문서를 많이 만드는 방식이 아니다.

여기서 SDD는 다음 의미를 가진다.

### 4.1 문서 우선
문서가 기준이다.  
구현은 문서를 따른다.

문서는 보고용 산출물이 아니라,  
구현 전에 의도와 범위를 정리하는 기준이다.

### 4.2 문서는 AI 협업의 입력 자산이다
문서는 사람이 보기 위한 설명서이면서 동시에,  
AI가 작업하기 위한 실행 입력이다.

### 4.3 상태 분리
정책 문서와 상태 문서를 분리한다.  
현재 상태와 다음 진입점은 별도 상태 문서에서 관리한다.

### 4.4 구조 우선
기능 추가보다 구조 경계와 책임 분리를 먼저 고정한다.

### 4.5 AI 협업 통제
AI는 문서와 현재 활성 범위를 기준으로 작업한다.  
최종 판단과 다음 단계 전환의 책임은 개발자에게 있다.

### 4.6 사람 확정이 필수다
AI가 보강한 문서는 초안 또는 보강안일 뿐이다.  
사람이 confirm 하기 전에는 기준 문서로 간주하지 않는다.

### 4.7 최소 범위 우선
현재 단계에 필요한 최소 범위부터 정리하고 구현한다.  
아직 검증되지 않은 미래 확장을 먼저 확정하지 않는다.

### 4.8 반복 가능성 우선
한 번만 통하는 방식보다  
여러 프로젝트에 반복 적용 가능한 운영 방식을 우선한다.

## 5. specdrive에서의 SDD 실행 흐름

butgo-specdrive는 작업 흐름을 핵심 작업 단계 `doc`, `dev`, 별도 운영 단계 `session`, 전달 단위 생성 단계 `git` 으로 나눈다.

### 5.1 문서 단계 (`doc`)
문서를 구현 가능한 상태로 만드는 단계다.

핵심 흐름:

- reinforce
- confirm
- history-save

기본 흐름:

1. 개발자 문서 초안 작성
2. AI 문서 보강
3. 개발자 검토
4. 문서 확정
5. history 저장

### 5.2 개발 단계 (`dev`)
확정된 문서를 실제 개발 작업 단위로 실행하는 단계다.

핵심 흐름:

- task-split
- phase
- cycle
- status

기본 흐름:

1. 확정된 문서를 기준으로 task 분해
2. 현재 phase 설정
3. 현재 cycle 설정
4. task 진행
5. 상태 갱신 및 검토

### 5.3 세션 단계 (`session`)
세션 복구와 세션 저장을 다루는 운영 단계다.

핵심 흐름 예:

- start
- save

기본 흐름:

1. 세션 시작 시 현재 상태와 진입점을 복구
2. 필요한 `doc` 또는 `dev` 작업 수행
3. 세션 종료 시 세션 메모와 다음 진입점 정리
4. 전달 메시지가 필요하면 `git` 단계로 넘김

### 5.4 Git 단계 (`git`)
브랜치명, Git commit message, PR 제목/설명 생성을 다루는 단계다.

현재 skill 흐름 예:

- `git-commit`
- `github-pr`

기본 흐름:

1. 현재 브랜치와 변경 파일 확인
2. 안전한 commit 제안 준비
3. 명시 승인 후에만 commit/push 수행
4. 명시 승인 후에만 GitHub PR 생성

즉 specdrive는  
**문서를 다듬는 흐름**과  
**문서를 기준으로 개발하는 흐름**을 분리해서 다루고,  
세션 운영 흐름도 별도 단계로 유지한다.

## 6. 이 저장소를 어떻게 보는가

이 저장소는 크게 두 축으로 이해할 수 있다.

### 6.1 specdrive
specdrive는 개발 스펙 문서를 기반으로  
AI 협업을 실행하는 도구 영역이다.

여기에는 다음이 포함된다.

- 세션 시작 규칙
- 세션 저장 규칙
- 브랜치명 생성 지원
- 문맥 복구
- 문서 묶음 기준
- 프롬프트 실행 흐름
- 문서 검토와 동기화 방식
- 구조 점검 방식
- AI 협업 운영 규칙
- repo-local Codex skill 구조
- Git branch / commit / PR 메시지 생성 보조
- history / phase / cycle 관리 방식

즉, specdrive는 **어떻게 협업할 것인가**를 다룬다.

### 6.2 projects
projects는 실제 애플리케이션의  
개발 스펙 문서 작업공간이다.

여기에는 다음이 포함된다.

- 요구사항
- 설계
- 구현 계획
- 상태
- 히스토리
- 애플리케이션별 결정 사항
- 검증용 실제 프로그램 또는 실제 프로젝트 문서

즉, projects는 **무엇을 만들 것인가**를 다룬다.

현재는 specdrive와 projects를 한 저장소에서 함께 운영하면서,  
문서 기반 AI 협업 방식이 실제 프로젝트에 유효한지 검증한다.

장기적으로는 specdrive와 projects가 분리 가능한 구조를 지향한다.

### 6.3 standards
`standards`는 `projects` 하위에서 공통으로 사용하는 개발 표준 문서군이다.

예:

- naming
- java coding
- db
- api
- package structure
- git policy

이 문서들은 개별 프로젝트보다 상위의 공통 기준으로 사용한다.

## 7. 현재 프로젝트 정의

specdrive는 다음을 위한 도구다.

- 개발 스펙 문서를 기준으로 AI 협업을 실행한다.
- 문서 단계와 실제 개발 단계를 분리한다.
- 세션 복구/저장과 Git/PR 메시지 생성을 별도 운영 단계로 둔다.
- 반복 가능한 작업 절차를 repo-local Codex skill로 정규화한다.
- 현재는 Codex와의 협업을 1차 기준으로 삼는다.

현재 specdrive는 SaaS 제품이라기보다 다음에 가깝다.

- 내부 엔진
- 작업 운영체계
- repo-local skill workflow
- AI 협업 실행 규칙 묶음

여기서 중요한 점은 specdrive가 preview 파일을 많이 만드는 도구가 아니라는 것이다.

현재 기준으로 specdrive의 핵심 개념은 다음에 가깝다.

- project 문서를 읽고 AI 협업 흐름을 실행한다.
- AI 보강안과 사람 판단을 분리한다.
- 의미 있는 문서 변경은 실제 project 문서에 적용한다.
- 적용 근거와 결과 문서는 `docs/history/projects/**` 아래에 문서대장처럼 남긴다.

즉 specdrive는  
**문서 초안 생성기**보다  
**문서 기반 AI 협업 운영체계 + 문서 이력 관리 흐름**으로 이해하는 편이 맞다.

## 8. 현재 방향

현재 specdrive는 다음 방향을 기준으로 한다.

- SaaS 외형보다 core workflow를 먼저 검증한다.
- 개발문서 단계와 실제 개발 단계를 분리한다.
- 현재는 Codex 중심으로 흐름을 정리한다.
- **repo-local Codex skill로 최소 작업 흐름을 먼저 검증한다.**
- **반복 검증된 흐름은 wizard 방식의 안내형 UX로 정리하는 것을 지향한다.**
- PowerShell CLI 작업은 현재 실행 중심이 아니라 후속 기술 후보로 둔다.
- 과거 단일 진입점 PowerShell CLI 라우터는 현재 기준에서 제거했다.
- 이후 필요 시 Go 또는 Python으로 승격을 검토한다.
- 현재는 멀티 AI 엔진 지원을 우선하지 않는다.

## 9. 현재 우선순위

현재 1차 우선순위는 다음과 같다.

### 문서 단계
- `doc draft-save`
- `doc reinforce-prompt`
- `doc reinforce`
- `doc confirm-prompt`
- `doc apply-prompt`
- `doc apply-only-prompt`
- 현재 해석 방향: prompt-first 보강 / 검토 / 반영 흐름

### 개발 단계
- `dev task-split`
- `dev phase set`
- `dev cycle set`
- `dev status`

### 세션 단계
- `$session start-lite`
- `$session start`
- `$session status`
- `$session save`

### Git 단계
- `$git-commit`
- `$github-pr`

현재는 웹 UI, 멀티테넌시, 조직/사용자 관리보다  
**문서 기반 AI 협업 흐름 자체를 안정화하는 것**이 더 중요하다.

## 10. 현재 개발 환경

현재 이 저장소의 최소 검증 환경은 다음을 기준으로 한다.

- 에디터: VS Code
- AI 협업 도구: Codex 확장
- 실행 인터페이스: `.agents/skills/**` 아래의 repo-local Codex skill
- 셸: 필요한 경우 로컬 Git 및 파일 확인에 사용하는 PowerShell

즉 현재 단계에서는  
**VS Code + Codex 확장 + repo-local Codex skill** 조합을 기본 개발 환경으로 본다.

직접 `codex exec` 연동과 넓은 범위의 CLI 자동화는 후속 범위로 둔다.
현재 기준에는 단일 진입점 `specdrive` CLI를 포함하지 않는다.

## 11. specdrive가 다루는 것

butgo-specdrive는 다음 흐름을 다룬다.

- 개발 스펙 문서 기반 협업
- 현재 상태 복구
- 세션 시작과 진입점 정리
- 세션 저장과 다음 진입점 보존
- 브랜치 / commit / PR workflow 지원
- AI 입력 문맥 구성
- 문서 검토와 보강
- 문서 변경 후 동기화
- 구조 점검과 경계 유지
- 작업 분해와 상태 관리
- Git / PR 메시지 생성 보조
- 반복 가능한 협업 흐름 정리

---

## 12. specdrive가 직접 소유하지 않는 것

butgo-specdrive는 다음을 직접 소유하는 시스템이 아니다.

- 개별 애플리케이션의 도메인 내용 자체
- 특정 프로젝트의 요구사항 본문 자체
- 특정 프로젝트의 세부 설계 판단 자체
- 모든 구현 결정을 대신 내려주는 자율형 AI
- 사람 검토가 완전히 제거된 자동 전환 시스템
- 현재 단계에서의 SaaS 웹 UI
- 조직 / 사용자 / 결제 기능
- 멀티 AI 엔진 일반화
- 운영 / 배포 자동화

즉, specdrive는 프로젝트 내용을 대신 작성하는 플랫폼이 아니라,  
**프로젝트 문서를 기반으로 AI 협업을 실행하고 운영하는 도구**다.

## 13. 포함 범위

현재 범위에 포함되는 것은 다음과 같다.

- 문서 보강 / 확정 / history 저장 흐름
- task 분해 흐름
- phase / cycle 상태 관리
- 프로젝트 문서와 공통 standards 분리
- repo-local Codex skill workflow 검증
- 실제 검증용 프로젝트를 통한 운영 방식 확인

## 14. 이 저장소의 의미

이 저장소는 단순히 specdrive만을 위한 저장소가 아니다.

현재는 다음을 함께 검증하는 작업 공간이다.

- specdrive 자체 문서와 skill workflow
- 공통 표준 문서 구조
- 실제 프로젝트 문서 작업공간 구조
- AI 협업이 실제 개발 프로젝트에 유효한지 여부

즉 현재 저장소는  
**specdrive 엔진과 실제 프로젝트 적용 방식을 함께 검증하는 작업 저장소**로 본다.

## 15. 어디서부터 읽으면 좋은가

이 저장소를 처음 보거나 다시 진입할 때는 아래 순서로 읽는 것이 좋다.

1. `AGENTS.md`  
   저장소 전체의 공통 작업 규칙과 협업 원칙

2. `docs/AI_CONTEXT.md`  
   현재 상태, 현재 focus, 다음 진입점

3. `docs/README.md`  
   docs 전체 구조와 진입 문서 안내

4. `docs/specdrive/README.md`  
   specdrive 자체의 정체성과 문서 구조

5. `docs/projects/README.md`  
   projects 영역의 역할과 구조

필요한 경우 이후에는 각 영역의 `AGENTS.md`, `README.md`, `index.md`와  
실제 작업 대상 문서를 따라가면 된다.

## 16. 현재 문서 구조

현재 문서 구조는 아래 축을 중심으로 운영된다.

### 16.1 공통 운영 기준
- `AGENTS.md`
  - 저장소 전체 공통 작업 규칙
- `docs/AI_CONTEXT.md`
  - 현재 상태 / 다음 진입점 / 세션 복구용 요약
- `docs/README.md`
  - docs 전체 진입 문서

### 16.2 specdrive 문서
- `docs/specdrive/**`
  - specdrive 자체의 운영 구조와 협업 도구 문서군

예:
- `docs/specdrive/AGENTS.md`
- `docs/specdrive/README.md`
- `docs/specdrive/index.md`

### 16.3 projects 문서
- `docs/projects/**`
  - 실제 애플리케이션의 개발 스펙 문서군

예:
- `docs/projects/README.md`

- `docs/projects/standards/**`

  - 현재는 projects를 보조하는 하위 공통 문서 성격으로 본다

예:
- `docs/projects/standards/index.md`

## 17. 현재 적용 방향

현재 butgo-specdrive는  
실제 애플리케이션 프로젝트의 개발 스펙 문서를 기반으로,  
AI 협업 흐름을 실행하는 도구 방향으로 정리되고 있다.

즉, 현재 방향은 다음과 같다.

- projects에 실제 애플리케이션 스펙 문서를 작성한다.
- 필요 시 projects는 공통 개발 기준 문서를 참조한다.
- specdrive는 이 문서들을 기반으로 세션 시작, 문맥 구성, 검토, 동기화, 작업 분해 같은 협업 흐름을 지원한다.

첫 번째 실제 애플리케이션 예시로는  
board 성격의 프로젝트를 두고 구조를 검증하는 방향을 고려하고 있다.

이때 board는 specdrive의 하위 기능이 아니라,  
**specdrive가 다루는 실제 애플리케이션 프로젝트**로 본다.

현재는 검증을 위해 하나의 저장소 안에 함께 두고 있지만,  
장기적으로는 specdrive와 개별 projects가 분리 가능한 구조를 지향한다.

## 18. 향후 방향

현재는 **core skill-first 운영 모델**을 먼저 검증한다.
반복 검증 이후의 제품 방향은 **wizard 방식의 안내형 흐름**이다.  
즉 사용자가 원시 명령을 기억하지 않아도 다음 action, 필요한 context, 사람 확인 지점, 기대 출력이 단계적으로 드러나는 구조를 지향한다.

이후 다음을 단계적으로 검토할 수 있다.

- 안정화된 `doc`, `session`, `git` 흐름의 wizard형 UX
- 구조 정리 및 폴더 재구성
- Go 또는 Python 기반 재구현
- Git 연계 강화
- 팀 협업 기능 강화
- 모바일 앱과 연계 가능한 협업 도구 검토
- 장기적으로 SaaS 형태 승격

단, 현재는 미래 확장보다 **지금 실제로 반복 가능한 흐름을 만드는 것**을 우선한다.
