# db-standard.md

## 1. 문서 목적

이 문서는 초기 게시판 프로젝트에서 사용할 데이터베이스 표준을 정의한다.

주요 목적은 다음과 같다.

- 테이블, 컬럼, 키, 인덱스 설계 기준을 통일한다.
- 자바 코드, API, DB 사이의 이름 불일치를 줄인다.
- 초기 레이어드 게시판 구현에서 흔들리기 쉬운 공통 컬럼, 상태값, 삭제 정책 기준을 먼저 고정한다.
- 이후 공통 모듈 분리 또는 헥사고날 전환 시에도 유지 가능한 최소 DB 규칙을 먼저 정리한다.
- AI 협업 시 DB 설계 판단 기준을 통일한다.

이 문서는 특정 DBMS 기능 전체를 설명하는 문서가 아니라,  
**프로젝트 전반에서 어떤 기준으로 테이블과 컬럼을 설계할지 정의하는 공통 표준 문서**이다.

---

## 2. 기본 원칙

이 프로젝트의 DB 설계는 다음 원칙을 따른다.

- 테이블과 컬럼 이름은 의미가 분명해야 한다.
- 자바 도메인 이름, API 이름, DB 이름은 가능한 한 같은 의미를 유지한다.
- 지금 필요한 범위를 넘는 과도한 일반화는 피한다.
- 업무 규칙을 모두 DB에 밀어 넣지 않는다.
- 반대로 데이터 무결성에 필요한 최소 제약은 DB에서도 유지한다.
- 읽기 쉬운 스키마가 복잡한 축약형 이름보다 우선한다.
- 상태값과 삭제 정책은 프로젝트 전반에서 일관되어야 한다.
- 공통 컬럼은 규칙을 먼저 정하고 반복 적용한다.
- 나중에 분리 가능한 구조를 염두에 두되, 처음부터 지나치게 모듈화하지 않는다.

---

## 3. 적용 범위

현재 단계에서는 초기 게시판 프로젝트를 기준으로 다음 범위에 적용한다.

- 게시판
- 게시글
- 댓글
- 사용자
- 첨부파일 메타데이터
- 권한/역할 관련 최소 테이블
- 공통 코드성 테이블이 필요할 경우 그 기준

현재 문서는 게시판 최소 구현에 바로 필요한 범위를 우선하며,  
향후 통계, 감사 로그, 이력 저장, 공통 모듈용 확장 테이블은 필요 시 별도 검토한다.

---

## 4. 테이블 이름 규칙

### 4.1 기본 규칙

- 테이블명은 소문자 snake_case를 사용한다.
- 단수형을 기본으로 한다.
- 의미가 분명한 명사를 사용한다.
- 축약어보다 전체 의미가 드러나는 이름을 우선한다.

예:
- `board`
- `post`
- `comment`
- `user_account`
- `attachment`
- `user_role`

지양 예:
- `posts`
- `tb_post`
- `pst`
- `cmnt`
- `usr`

---

### 4.2 복합 의미 테이블

두 개 이상의 의미가 결합된 경우, 앞에서 뒤로 자연스럽게 읽히는 이름을 사용한다.

예:
- `board_category`
- `post_attachment`
- `user_permission`
- `role_permission`

---

### 4.3 조인 성격 테이블

다대다 연결이나 관계 표현용 테이블은 두 개념을 모두 드러낸다.

예:
- `user_role`
- `role_permission`
- `post_tag`

현재 단계에서 꼭 필요하지 않다면 조인 테이블을 미리 일반화하지 않는다.

---

## 5. 컬럼 이름 규칙

### 5.1 기본 규칙

- 컬럼명은 소문자 snake_case를 사용한다.
- 컬럼명은 의미 중심으로 작성한다.
- 자바 필드명과 가능한 한 같은 의미를 유지한다.

예:
- `post_id`
- `board_id`
- `title`
- `content`
- `author_user_id`
- `created_at`
- `updated_at`

지양 예:
- `postid`
- `ttl`
- `cnts`
- `crt_dt`
- `upd_dtm`

---

### 5.2 날짜/시간 컬럼 규칙

날짜/시간 컬럼은 의미가 드러나게 작성한다.

예:
- `created_at`
- `updated_at`
- `deleted_at`
- `published_at`

지양 예:
- `reg_date`
- `mod_dt`
- `del_yn_date`

---

### 5.3 수량/정렬/상태 컬럼 규칙

의미가 명확히 드러나야 한다.

예:
- `comment_count`
- `view_count`
- `sort_order`
- `status`

지양 예:
- `cnt`
- `ord`
- `flag`

---

## 6. PK 규칙

### 6.1 기본 PK 컬럼명

PK 컬럼은 `<table>_id` 형식을 기본으로 한다.

예:
- `board_id`
- `post_id`
- `comment_id`
- `user_id`

---

### 6.2 PK 타입

현재 단계에서는 `bigint` 기반의 단일 surrogate key를 기본으로 한다.

기본 원칙:
- PK는 숫자형 단일 키를 우선한다.
- 복합 PK는 특별한 이유가 없으면 사용하지 않는다.
- 자연키는 업무상 의미가 있더라도 PK보다 unique key 후보로 먼저 검토한다.

---

### 6.3 PK 생성 전략

- 자동 증가 또는 그에 준하는 일관된 생성 전략을 사용한다.
- 프로젝트 전체에서 PK 생성 방식은 가능한 한 통일한다.
- 같은 프로젝트 안에서 테이블마다 서로 다른 생성 전략을 무분별하게 섞지 않는다.

---

## 7. FK 규칙

### 7.1 FK 컬럼명

FK 컬럼도 참조 대상을 드러내는 이름을 사용한다.

예:
- `board_id`
- `post_id`
- `author_user_id`
- `parent_comment_id`

---

### 7.2 FK 사용 원칙

- FK는 의미상 명확한 관계에는 우선적으로 사용한다.
- FK를 생략할 특별한 이유가 있으면 문서나 코드에서 명확히 남긴다.
- 삭제 정책과 FK 제약이 충돌하지 않도록 미리 정한다.

---

### 7.3 자기참조 FK

계층형 구조가 필요한 경우 자기참조 FK를 사용할 수 있다.

예:
- `parent_comment_id`

현재 단계에서는 댓글 계층 구조가 실제 요구될 때만 도입한다.

---

## 8. 공통 컬럼 규칙

### 8.1 기본 공통 컬럼

대부분의 주요 테이블은 아래 공통 컬럼을 기본으로 검토한다.

- `created_at`
- `updated_at`

필요 시:
- `deleted_at`
- `created_by`
- `updated_by`

---

### 8.2 공통 컬럼 사용 원칙

- 생성 시각과 수정 시각은 가능하면 주요 업무 테이블에 공통 적용한다.
- 사용자 추적 컬럼은 인증/감사 요구가 명확해질 때 도입한다.
- 지금 단계에서 필요 없는 공통 컬럼을 과도하게 미리 추가하지 않는다.

---

## 9. 삭제 정책 규칙

### 9.1 기본 원칙

초기 단계에서는 **soft delete를 기본 전제처럼 무조건 도입하지 않는다.**

삭제 정책은 아래 기준으로 판단한다.

- 실제 복구 요구가 있는가
- 조회에서 삭제 제외 처리가 복잡해지지 않는가
- 이력/감사 요구가 있는가

---

### 9.2 hard delete / soft delete 기준

#### hard delete를 우선 검토하는 경우
- 단순 게시판 최소 구현
- 복구 요구가 아직 없음
- 삭제 데이터 보존 요구가 없음

#### soft delete를 검토하는 경우
- 삭제 복구 요구가 있음
- 감사 로그 또는 이력 요구가 있음
- 운영상 삭제 추적이 필요함

---

### 9.3 soft delete 컬럼 규칙

soft delete를 사용할 경우 `deleted_yn`보다 `deleted_at`을 우선 검토한다.

예:
- `deleted_at`

이유:
- 삭제 여부와 삭제 시점을 함께 표현 가능
- 추가적인 감사 의미를 담기 쉬움

지양:
- `del_yn`
- `use_yn`
- 의미가 불분명한 다중 플래그 남용

---

## 10. 상태값 규칙

### 10.1 상태 컬럼 기본 원칙

상태는 가능한 한 하나의 의미 있는 컬럼으로 표현한다.

예:
- `status`

상태값은 자바 Enum과 대응되도록 설계하는 것을 우선 검토한다.

예:
- `DRAFT`
- `PUBLISHED`
- `DELETED`

---

### 10.2 코드값/플래그 사용 기준

아래를 지양한다.

- `yn` 플래그를 무분별하게 늘리는 것
- 같은 개념을 여러 플래그로 분산하는 것
- 상태를 숫자 코드로만 숨기는 것

지양 예:
- `del_yn`
- `open_yn`
- `pub_yn`

더 나은 예:
- `status`
- `deleted_at`

---

## 11. 문자열 / 숫자 / 텍스트 타입 기준

### 11.1 문자열

- 길이가 명확한 식별값, 제목, 이름은 `varchar` 계열을 사용한다.
- 길이를 예측하기 어려운 본문은 text 계열을 검토한다.
- 무작정 긴 varchar 하나로 모든 문자열을 처리하지 않는다.

예:
- 제목: `varchar`
- 본문: `text`

---

### 11.2 숫자

- PK/FK는 `bigint`를 기본으로 한다.
- 개수, 순서, 상태 우선순위 등은 적절한 정수형을 사용한다.
- 금액이나 정밀 숫자가 필요한 경우 별도 기준을 추가한다.

현재 게시판 범위에서는 count, sort, id 중심의 정수형 기준이면 충분하다.

---

### 11.3 boolean

DB boolean 타입을 사용할 수 있으면 우선 검토한다.  
다만 상태가 늘어날 가능성이 있는 경우 boolean보다 상태 컬럼이 더 적절할 수 있다.

예:
- 단순 고정 여부: `pinned`
- 공개/비공개/삭제 등 복수 상태: `status`

---

## 12. 인덱스 규칙

### 12.1 기본 원칙

인덱스는 조회 패턴을 기준으로 추가한다.  
관성적으로 모든 컬럼에 인덱스를 걸지 않는다.

---

### 12.2 기본 검토 대상

아래는 인덱스 검토 대상이 된다.

- FK 컬럼
- 목록 조회의 정렬 기준 컬럼
- 자주 조회되는 상태 컬럼
- 검색 조건에 반복적으로 쓰이는 컬럼

예:
- `board_id`
- `author_user_id`
- `created_at`
- `status`

---

### 12.3 복합 인덱스

복합 인덱스는 실제 조회 패턴이 보일 때 도입한다.  
지금 단계에서는 미리 과하게 넣지 않는다.

예:
- `(board_id, status, created_at)`

---

## 13. Unique 제약 규칙

- 업무적으로 중복되면 안 되는 값에는 unique 제약을 검토한다.
- 자연키 성격이 있더라도 PK보다 unique 제약 후보로 먼저 본다.
- 단, 현재 요구가 불분명하면 과도한 unique 제약을 먼저 넣지 않는다.

예:
- 게시판 코드
- 사용자 로그인 ID
- 첨부파일 외부 식별자

---

## 14. 게시판 초기 테이블 설계 기준

현재 단계에서는 아래 정도를 기본 후보로 본다.

- `board`
- `post`
- `comment`
- `user_account`
- `attachment`
- 필요 시 `post_attachment`
- 필요 시 `role`
- 필요 시 `permission`
- 필요 시 `role_permission`
- 필요 시 `user_role`

이 목록은 지금 단계의 설계 출발점이며, 실제 범위에 따라 줄일 수 있다.

---

## 15. 자바 / API와의 대응 원칙

DB 이름은 자바와 API 이름과 가능한 한 같은 의미를 유지한다.

예:

- 자바 클래스: `PostEntity`
- API 경로: `/posts`
- DB 테이블: `post`

- 자바 필드: `createdAt`
- JSON 필드: `createdAt`
- DB 컬럼: `created_at`

즉, 형식은 다를 수 있지만 의미는 같아야 한다.

---

## 16. 금지 또는 지양 규칙

다음을 지양한다.

- `tb_`, `mst_`, `dtl_` 같은 관성적 접두사 남용
- 의미 없는 축약 컬럼명
- 모든 상태를 `yn` 플래그로 처리하는 방식
- 복합 PK 남용
- 조회 패턴 근거 없는 과도한 인덱스
- 자바/API/DB 이름이 서로 다른 용어를 쓰는 것
- soft delete를 이유 없이 전 테이블에 일괄 적용하는 것
- 지금 필요 없는 일반화 테이블을 미리 만드는 것

---

## 17. 현재 단계 적용 기준

현재 단계에서는 레이어드 게시판을 빠르게 시작하되,  
나중에 공통 모듈 분리 또는 구조 전환이 가능하도록 최소한의 일관성을 유지하는 것을 목표로 한다.

따라서 현재는 다음을 우선한다.

- 단수형 소문자 snake_case 테이블명 사용
- `<table>_id` 형식의 PK/FK 사용
- `created_at`, `updated_at` 중심의 공통 컬럼 적용
- `deleted_yn`보다 `deleted_at` 우선 검토
- `yn` 플래그 남용보다 의미 있는 상태 컬럼 우선
- 자바/API/DB 용어 일치 유지
- 필요한 범위 안에서만 인덱스와 제약 추가

즉, 현재 단계의 DB 표준은  
미래 확장 구조를 미리 완성하는 것이 아니라,  
**초기 게시판 구현이 흔들리지 않도록 하는 최소한의 공통 설계 기준**을 먼저 고정하는 데 목적이 있다.
