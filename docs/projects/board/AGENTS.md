# docs/projects/board/AGENTS.md

## 1. 문서 목적

이 문서는 board 프로젝트 전용 **작업 규칙**을 정의한다.

루트 `AGENTS.md` 가 저장소 전체의 최상위 작업 기준 문서(공통 헌법)이고,  
`docs/specdrive/AGENTS.md` 가 specdrive 자체 운영 규칙이라면,  
이 문서는 그 위에서 **board 애플리케이션 문서를 설계·작성·검증할 때의 추가 규칙**을 다룬다.

현재는 specdrive의 운영 방식을 검증하기 위해 같은 저장소 안에서 함께 운영하지만,  
장기적으로는 board가 specdrive와 분리 가능한 독립 애플리케이션 프로젝트 문서군이 되는 방향을 전제로 한다.

이 문서는 다음에 집중한다.

- board 문서군의 역할과 범위
- board 요구사항 / 설계 / 구현 계획 문서의 책임 분리
- specdrive 문서와 board 문서의 경계 유지
- board를 실제 애플리케이션 프로젝트 문서군으로 다루기 위한 원칙
- board 문서 작성 시 금지/주의 사항

---

## 2. 적용 범위

이 문서는 다음 범위에 적용된다.

- `docs/projects/board/**`
- board 요구사항, 설계, 구현 계획, 상태, 히스토리 관련 문서
- board 프로젝트 전용 구조와 정책
- board 프로젝트와 직접 연결되는 구현/검증 문서

specdrive 자체의 운영 규칙과 협업 흐름은 이 문서가 아니라  
`docs/specdrive/**` 에서 다룬다.

루트 공통 원칙은 `AGENTS.md` 를 따르고,  
specdrive 도구 자체의 규칙은 `docs/specdrive/AGENTS.md` 를 따른다.

---

## 3. board 프로젝트의 현재 성격

board는 specdrive의 하위 기능이 아니라,  
**specdrive가 다루는 실제 애플리케이션 프로젝트**다.

현재 board는 다음 목적을 가진다.

- specdrive의 문서 기반 AI 협업 방식이 실제 프로젝트에도 유효한지 검증한다.
- 레이어드 구조 기반의 예시 애플리케이션으로 문서 흐름을 검증한다.
- 실제 요구사항 / 설계 / 구현 계획 / 상태 / 히스토리 문서군을 운영해 본다.
- 필요 시 이후 다른 프로젝트에도 적용 가능한 문서 작성 및 협업 기준을 점검한다.

현재 단계에서 board는 제품 완성보다  
**문서 기반 개발 흐름 검증과 실제 적용 가능성 확인**에 더 큰 의미를 가진다.

---

## 4. board 문서의 역할 원칙

board 문서는 실제 애플리케이션 개발 내용을 다루어야 한다.

예:

- 요구사항
- 설계
- 구현 계획
- 상태
- 히스토리
- 결정 사항
- 기능 단위 작업 문서

board 문서는 specdrive 운영 규칙을 설명하는 문서가 아니다.  
즉 board 문서 안에는 다음이 중심이 되어야 한다.

- board가 무엇을 만드는지
- 어떤 구조로 만들지
- 어떤 제약과 경계를 가질지
- 어떤 순서로 구현할지
- 현재 어떤 상태인지

반대로 다음은 board 문서의 중심 내용이 아니다.

- specdrive 자체 CLI 설계
- specdrive의 skill 구조
- Codex exec 상세 설계
- specdrive 전용 협업 절차 설계
- 저장소 전체 공통 운영 규칙

이런 내용은 specdrive 문서나 루트 문서에서 다룬다.

---

## 5. specdrive와 board의 분리 원칙

### 5.1 specdrive는 협업 방식을 다룬다
specdrive는 다음을 다룬다.

- 세션 시작
- 문맥 복구
- 문서 보강 절차
- 문서 확정 절차
- history 저장 절차
- task 분해 절차
- phase / cycle 상태 관리
- 명령 실행 흐름
- AI 협업 규칙

### 5.2 board는 실제 애플리케이션 내용을 다룬다
board는 다음을 다룬다.

- 도메인 요구사항
- 화면/기능 흐름
- 구조 결정
- 패키지/레이어 구성
- API/DB 정책의 적용
- 구현 범위와 단계
- 기능별 작업 계획
- 현재 진행 상태와 판단 이력

### 5.3 혼합 금지
board 문서에 specdrive 자체 운영 규칙을 과하게 넣지 않는다.  
반대로 specdrive 문서에 board 상세 설계를 넣지 않는다.

---

## 6. board 개발 문서 원칙

### 6.1 문서가 기준이다
문서가 기준이고, 구현은 문서를 따른다.

### 6.2 현재 결정 우선
가능성 나열보다 현재 채택한 설계와 범위를 먼저 기록한다.

### 6.3 요구사항 / 설계 / 구현 계획 / 상태 / 히스토리를 분리한다
하나의 문서가 너무 많은 역할을 동시에 맡지 않게 한다.

예:

- 요구사항 문서
- 구조/설계 문서
- 구현 계획 문서
- 상태 문서
- 히스토리 문서

### 6.4 최소 범위부터 시작한다
현재 검증 단계에 필요한 최소 범위부터 문서화하고 구현한다.  
미래 확장을 이유로 현재 구조를 과도하게 복잡하게 만들지 않는다.

### 6.5 AI가 읽기 쉬운 구조를 유지한다
board 문서는 AI 협업 입력으로도 사용되므로 다음을 유지한다.

- 제목 체계 일관성
- 용어 일관성
- 현재 범위와 후속 범위 구분
- 관련 문서 명시
- 현재 상태와 보류 사항 구분

## 7. board 작업 흐름 원칙

board도 전체 흐름에서는 `doc` 단계와 `dev` 단계를 따른다.

### 7.1 doc 단계
board 문서를 구현 가능한 상태로 만드는 단계다.

핵심 흐름:

- reinforce
- confirm
- history-save

원칙:

- board 문서 초안이 먼저 있어야 한다.
- AI는 현재 문서를 기준으로 보강한다.
- 사람이 confirm 하기 전에는 기준 문서가 아니다.
- 중요한 구조 판단은 history로 남긴다.

### 7.2 dev 단계
confirm 된 board 문서를 기준으로 실제 개발 작업으로 넘어가는 단계다.

핵심 흐름:

- task-split
- phase
- cycle
- status

원칙:

- 구현은 confirm 된 문서를 기준으로 진행한다.
- task-split은 board 기능/구조 문서를 기준으로 수행한다.
- phase / cycle은 board 구현의 현재 단계와 검증 수준을 나타낸다.
- dev 단계가 문서 미확정 상태를 덮어쓰지 않도록 한다.

---

## 8. board의 현재 구조 방향

현재 board는 **Spring Boot 기반 레이어드 구조**를 우선 검증 대상으로 본다.

원칙:

- 처음부터 과한 구조 확장을 하지 않는다.
- 현재는 레이어드 구조를 기준으로 문서와 구현을 정리한다.
- 헥사고날 전환은 필요가 검증된 후 후속 검토 대상으로 둔다.
- 공통 standards 문서를 참조하되, board 문서 안에서 실제 적용 판단을 명확히 적는다.

즉 board 문서는 추상적인 구조 이상론보다  
**현재 채택한 구조와 그 이유**를 더 분명히 담아야 한다.

---

## 9. 공통 standards 적용 원칙

board는 필요 시 `docs/projects/standards/**` 의 공통 기준을 참조한다.

예:

- naming
- java coding
- db
- api
- package structure
- git policy

원칙:

- standards는 공통 기준이다.
- board 문서는 그 기준을 board에 어떻게 적용하는지를 다룬다.
- 공통 규칙 자체를 board 문서에 중복해서 길게 복사하지 않는다.
- board에서 예외나 추가 판단이 있으면 board 문서에 명시한다.

## 10. 금지 및 주의 사항

### 10.1 금지
다음을 먼저 제안하거나 문서에 확정처럼 쓰지 말 것.

- 아직 검증되지 않은 장기 구조를 현재 구조처럼 기록하는 것
- board를 specdrive 자체 기능처럼 서술하는 것
- specdrive 운영 규칙을 board 문서 중심 내용으로 끌어오는 것
- 문서 확정 전에 구현 상세를 과도하게 밀어넣는 것
- 공통 standards와 충돌하는 board 고유 규칙을 근거 없이 만드는 것

### 10.2 주의
다음을 특히 주의한다.

- board는 실제 애플리케이션 프로젝트 문서군이어야 한다.
- board 문서는 현재 구현 범위와 후속 후보를 명확히 구분해야 한다.
- 검증용 프로젝트라는 이유로 문서 품질 기준을 낮추지 않는다.
- 단기 편의를 위해 구조 경계를 흐리지 않는다.

## 11. 개발자와 AI의 역할 원칙

### 11.1 개발자
개발자는 다음 책임을 가진다.

- board의 현재 범위와 우선순위를 결정한다.
- 문서와 구현의 최종 승인 책임을 가진다.
- 구조 경계와 현재 단계 전환 시점을 판단한다.
- AI 결과를 채택할지, 수정할지, 보류할지 결정한다.

### 11.2 AI
AI는 다음 역할을 수행한다.

- board 문서 초안 보강
- board 문서 구조/중복/충돌 검토
- board 구현 보조
- 테스트/정리/설명 보조

AI는 다음을 해서는 안 된다.

- 명시되지 않은 기능/구조 결정을 확정처럼 다루는 것
- 현재 board 범위를 넘는 구현을 먼저 진행하는 것
- board를 specdrive 내부 기능처럼 다루는 것
- 기존 문서 역할을 임의로 합치거나 삭제하는 것

---

## 12. 권장 작업 흐름

board 문서를 다룰 때의 기본 권장 흐름은 다음과 같다.

1. 현재 대상이 board 문서가 맞는지 먼저 확인한다.
2. 루트 `AGENTS.md` 와 이 문서를 먼저 확인한다.
3. 관련 standards가 있으면 함께 확인한다.
4. 현재 문서를 초안 또는 수정한다.
5. AI로 보강 또는 검토한다.
6. 개발자가 최종 판단한다.
7. 필요한 경우 연관 문서와 상태 문서를 함께 반영한다.
8. 중요한 변경은 history를 남긴다.

## 13. 이 문서가 직접 다루는 것과 다루지 않는 것

### 이 문서가 직접 다루는 것
- board 전용 작업 규칙
- board 문서군의 역할/경계
- board와 specdrive의 분리 기준
- board 문서 작성과 검토 원칙
- board 문서의 현재 구조 방향
- standards 적용 방식의 전용 기준

### 이 문서가 직접 다루지 않는 것
- 저장소 전체 공통 규칙
- specdrive 자체 운영 규칙
- specdrive CLI/skill/Codex 연동 상세 설계
- 다른 프로젝트 전용 규칙
- 현재 시점의 구체적인 활성 상태 자체

이런 내용은 루트 문서, specdrive 문서, 상태 문서, 개별 문서에서 다룬다.

---

## 14. 최종 원칙

board의 핵심은  
specdrive를 설명하는 것이 아니라,  
**실제 애플리케이션 프로젝트 문서를 올바르게 작성하고 유지하는 것**이다.

따라서 board를 다룰 때는 항상 다음을 먼저 확인한다.

- 이 내용은 board 자체 판단인가?
- 이 내용은 specdrive 공통 규칙인가?
- 이 내용은 standards 공통 기준인가?
- 지금 필요한 최소 구조를 강화하는가?
- 현재 범위와 후속 범위를 분리해서 기록하고 있는가?

이 질문에 답하지 못하면,  
기능 추가보다 문서 위치와 역할부터 다시 정리한다.
