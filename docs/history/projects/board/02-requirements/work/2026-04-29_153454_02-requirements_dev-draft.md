# specs/02-requirements.md

## 1. 문서 목적

이 문서는 board 프로젝트의 요구사항 문서다.

현재 단계에서 board 프로젝트가 무엇을 만들어야 하는지,  
어떤 범위를 포함하고 제외하는지,  
후속 설계 문서와 구현 계획 문서가 받아야 할 입력을 정리한다.

이 문서는 상세 설계 문서가 아니다.  
API 상세, DB 상세, 패키지 구조, 구현 순서는 후속 문서에서 다룬다.

현재 문서의 목적은 다음과 같다.

- board의 현재 단계 최소 기능 요구사항을 정의한다.
- 게시판 기본 CRUD 흐름을 요구사항 관점에서 고정한다.
- 문서 기반 개발 흐름 검증에 필요한 요구사항을 정리한다.
- 후속 설계 문서가 구조와 책임을 나눌 수 있도록 기준을 제공한다.
- 구현 계획 문서가 작업 단위를 나눌 수 있도록 입력을 제공한다.

---

## 2. 프로젝트 요구사항 한 줄 요약

board는 specdrive 방식으로 실제 개발 흐름을 검증하기 위한  
Spring Boot 기반 layered 게시판 애플리케이션이다.

현재 단계의 핵심 요구사항은  
게시글 작성, 조회, 수정, 삭제의 최소 흐름을 구현 가능한 수준으로 정의하는 것이다.

---

## 3. 요구사항 정리 기준

현재 요구사항은 다음 기준으로 정리한다.

- 현재 단계에서 구현과 검증이 가능한 최소 범위를 우선한다.
- 게시판 제품 전체가 아니라 최소 게시판 애플리케이션을 기준으로 한다.
- 상세 설계, API 명세, DB 설계, 패키지 구조를 이 문서에서 확정하지 않는다.
- Spring Boot 기반 layered 구조를 전제로 하되, 구조 상세는 설계 문서에서 다룬다.
- 미래 확장을 이유로 현재 요구사항을 과도하게 넓히지 않는다.
- 현재 범위와 후속 후보를 명확히 구분한다.
- 문서 우선 개발 흐름을 검증할 수 있는 수준으로 요구사항을 정리한다.

---

## 4. 현재 목표

현재 board 프로젝트의 요구사항 목표는 다음과 같다.

- 게시판 기본 CRUD 흐름을 정의한다.
- Spring Boot 기반 layered 애플리케이션으로 구현 가능한 최소 기능 범위를 정한다.
- 요구사항에서 설계, 구현 계획, 현재 상태 문서로 이어지는 흐름을 검증한다.
- logging, auth, board 같은 재사용 후보 영역의 경계를 검토할 수 있는 기준을 남긴다.
- 후속 확장을 고려하되 현재 구현 범위를 복잡하게 만들지 않는다.

현재 단계에서 board는 완성형 게시판 제품이 아니다.  
문서 기반 AI 협업 흐름과 최소 애플리케이션 구조를 검증하기 위한 대상 프로젝트다.

---

## 5. 현재 포함 범위

현재 단계에서 포함하는 요구사항 범위는 다음과 같다.

- 게시글 목록 조회
- 게시글 상세 조회
- 게시글 등록
- 게시글 수정
- 게시글 삭제
- 게시글 기본 데이터 관리
- 최소 입력 검증
- Spring Boot 기반 layered 구조 적용을 위한 요구사항 정리
- 문서 기반 개발 흐름 검증에 필요한 최소 문서 연계
- logging 등 공통 기능의 적용 가능성 검토
- auth와 board의 경계 검토

현재는 게시판의 기본 CRUD 흐름을 첫 번째 기능 범위로 본다.

---

## 6. 현재 제외 범위

현재 단계에서 직접 포함하지 않는 범위는 다음과 같다.

- 헥사고날 구조 전제의 문서 체계 확정
- auth와 board의 독립 배포 구조 확정
- 멀티테넌시
- 대규모 검색, 통계, 분석 기능
- 과도한 공통 모듈 일반화
- 공통 모듈의 완성형 제품화
- 댓글, 대댓글
- 첨부파일
- 좋아요 / 추천
- 사용자 회원가입
- 사용자 로그인
- 상세 권한 관리
- 관리자 기능
- 모바일 앱 구조
- 상세 API 명세 확정
- 상세 DB 설계 확정
- 상세 패키지 구조 확정

현재 단계에서는 미래 확장을 이유로 현재 구조와 요구사항을 과도하게 복잡하게 만들지 않는다.

## 7. 기능 요구사항

### 7.1 게시글 목록 조회

사용자는 게시글 목록을 조회할 수 있어야 한다.

현재 단계의 최소 요구사항은 다음과 같다.

- 게시글 목록을 확인할 수 있어야 한다.
- 목록 항목에는 게시글을 구분할 수 있는 정보가 포함되어야 한다.
- 목록 항목에는 제목이 포함되어야 한다.
- 목록에서 상세 조회 흐름으로 이어질 수 있어야 한다.
- 목록 조회는 게시글 기본 CRUD 흐름의 진입점 역할을 해야 한다.

현재 문서에서는 페이징, 정렬, 검색 조건을 상세 확정하지 않는다.  
필요 시 API 명세 또는 설계 문서에서 최소 기준을 정한다.

### 7.2 게시글 상세 조회

사용자는 게시글 하나의 상세 내용을 조회할 수 있어야 한다.

현재 단계의 최소 요구사항은 다음과 같다.

- 게시글 식별자를 기준으로 단건 게시글을 조회할 수 있어야 한다.
- 상세 화면 또는 상세 응답에서 제목을 확인할 수 있어야 한다.
- 상세 화면 또는 상세 응답에서 본문을 확인할 수 있어야 한다.
- 상세 조회 후 수정 또는 삭제 흐름으로 이어질 수 있어야 한다.
- 존재하지 않는 게시글을 조회할 경우 적절히 처리되어야 한다.

오류 처리 방식과 응답 형식은 이 문서에서 확정하지 않는다.  
후속 API 명세와 설계 문서에서 구체화한다.

### 7.3 게시글 등록

사용자는 새 게시글을 등록할 수 있어야 한다.

현재 단계의 최소 요구사항은 다음과 같다.

- 제목을 입력할 수 있어야 한다.
- 본문을 입력할 수 있어야 한다.
- 제목은 필수 입력값으로 본다.
- 본문은 필수 입력값으로 본다.
- 등록이 완료되면 목록 또는 상세 조회 흐름으로 이어질 수 있어야 한다.
- 등록된 게시글은 이후 목록과 상세 조회에서 확인할 수 있어야 한다.

작성자, 권한, 로그인 사용자 연계는 현재 단계에서 확정하지 않는다.

### 7.4 게시글 수정

사용자는 기존 게시글을 수정할 수 있어야 한다.

현재 단계의 최소 요구사항은 다음과 같다.

- 기존 게시글의 제목을 수정할 수 있어야 한다.
- 기존 게시글의 본문을 수정할 수 있어야 한다.
- 수정 시 제목은 비어 있으면 안 된다.
- 수정 시 본문은 비어 있으면 안 된다.
- 수정이 완료되면 수정된 내용을 확인할 수 있어야 한다.
- 존재하지 않는 게시글을 수정하려는 경우 적절히 처리되어야 한다.

수정 권한과 사용자 인증 연계는 현재 단계에서 확정하지 않는다.

### 7.5 게시글 삭제

사용자는 기존 게시글을 삭제할 수 있어야 한다.

현재 단계의 최소 요구사항은 다음과 같다.

- 삭제 대상 게시글을 식별할 수 있어야 한다.
- 게시글을 삭제할 수 있어야 한다.
- 삭제 후 목록 조회 흐름으로 이어질 수 있어야 한다.
- 삭제된 게시글은 일반 조회 흐름에서 다시 조회되지 않아야 한다.
- 존재하지 않는 게시글을 삭제하려는 경우 적절히 처리되어야 한다.

논리 삭제와 물리 삭제 중 어떤 방식을 사용할지는 이 문서에서 확정하지 않는다.  
DB 설계 또는 설계 문서에서 필요 시 결정한다.

---

## 8. 데이터 요구사항

현재 단계에서 게시글은 최소한 다음 정보를 가진다고 본다.

- 식별자
- 제목
- 본문
- 생성 시각
- 수정 시각

현재 문서에서는 데이터 구조를 상세 설계하지 않는다.  
테이블명, 컬럼명, 제약 조건, 인덱스, 마이그레이션 방식은 DB 설계 문서에서 다룬다.

---

## 9. 입력 검증 요구사항

현재 단계에서 필요한 최소 입력 검증 요구사항은 다음과 같다.

- 제목은 비어 있으면 안 된다.
- 본문은 비어 있으면 안 된다.
- 잘못된 입력에 대해 사용자가 다시 시도할 수 있어야 한다.
- 잘못된 입력은 정상 등록 또는 정상 수정으로 처리되면 안 된다.

검증 방식, 에러 응답 형식, 예외 처리 구조는 이 문서에서 확정하지 않는다.  
후속 설계 문서와 API 명세 문서에서 구체화한다.

---

## 10. 구조 요구사항

현재 board는 Spring Boot 기반 layered 구조를 우선한다.

현재 단계의 구조 요구사항은 다음과 같다.

- 게시판 기능은 layered 구조로 구현 가능한 형태여야 한다.
- Controller, Service, Repository 같은 일반적인 Spring Boot 계층 분리를 고려한다.
- 상세 패키지 구조는 애플리케이션 구조 문서에서 정리한다.
- JPA 기반 persistence 적용을 우선 검토한다.
- Flyway 사용 가능성은 DB 설계 문서에서 검토한다.
- 현재 단계에서는 헥사고날 구조를 전제로 하지 않는다.
- 미래 확장을 이유로 현재 구조를 과도하게 복잡하게 만들지 않는다.

---

## 11. 공통 기능 및 재사용 후보 요구사항

현재 board는 하나의 애플리케이션으로 시작한다.

다만 다음 영역은 후속 재사용 가능성을 고려한다.

- logging
- auth
- board

현재 단계의 요구사항은 다음과 같다.

- logging은 공통 기능 후보로 본다.
- auth는 현재 단계에서 상세 구현 범위를 확정하지 않는다.
- board는 현재 애플리케이션의 핵심 기능 영역으로 본다.
- logging, auth, board의 경계는 구조 문서에서 검토한다.
- 공통 모듈을 완성형 제품으로 일반화하지 않는다.
- starter 분리 가능성은 후속 구조 검토 대상으로 둔다.

---

## 12. 문서 기반 개발 흐름 요구사항

board 프로젝트는 specdrive 방식의 실제 적용 대상이다.

따라서 기능 요구사항 외에 다음 문서 흐름도 현재 요구사항에 포함한다.

- 요구사항 문서는 overview 문서의 범위와 방향을 따라야 한다.
- 설계 문서는 요구사항 문서를 입력으로 받아야 한다.
- 애플리케이션 구조 문서는 설계 문서의 구조 판단을 구체화해야 한다.
- 도메인 모델 문서는 요구사항의 핵심 개념을 정리해야 한다.
- API 명세 문서는 요구사항의 입출력 흐름을 구체화해야 한다.
- DB 설계 문서는 요구사항의 데이터 항목을 구체화해야 한다.
- 구현 계획 문서는 요구사항과 설계를 기준으로 작업 단위를 나누어야 한다.
- 현재 상태 문서는 현재 활성 작업과 다음 진입점을 추적해야 한다.

현재 단계에서는 문서 간 역할이 섞이지 않도록 유지하는 것이 중요하다.

---

## 13. 비기능 요구사항

현재 단계에서 최소한 확인해야 하는 비기능 요구사항은 다음과 같다.

- 요구사항, 설계, 구현 계획 문서의 역할이 섞이지 않아야 한다.
- AI와 사람이 함께 검토하기 쉬운 문서 구조를 유지해야 한다.
- 현재 범위와 후속 후보가 구분되어야 한다.
- 작은 범위라도 실제 구현과 검증이 가능해야 한다.
- Spring Boot 기반 애플리케이션으로 자연스럽게 구현 가능해야 한다.
- 과도한 추상화보다 현재 범위의 명확성을 우선해야 한다.

성능, 보안, 운영, 확장성 목표는 현재 단계에서 상세 수치로 확정하지 않는다.  
필요 시 후속 문서에서 별도 기준을 정한다.
