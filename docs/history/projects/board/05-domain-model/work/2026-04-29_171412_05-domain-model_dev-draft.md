# specs/05-domain-model.md

## 1. 문서 목적

이 문서는 board 프로젝트의 도메인 모델 문서다.

현재 단계에서 게시판의 핵심 도메인 개념인 게시글이 어떤 속성과 규칙을 가지는지 정리한다.

이 문서는 상세 DB 설계 문서가 아니다.  
또한 상세 API 명세 문서나 구현 계획 문서도 아니다.

현재 문서의 목적은 다음과 같다.

- 요구사항 문서에서 정의한 게시글 CRUD 흐름의 핵심 개념을 정리한다.
- 설계 문서에서 정의한 layered 구조 안에서 사용할 도메인 기준을 정리한다.
- 게시글의 최소 속성과 기본 규칙을 정의한다.
- Entity, DTO, DB 설계가 이어받을 기준을 제공한다.
- 구현 계획 문서가 도메인 관련 작업 단위를 나눌 수 있도록 기준을 제공한다.

---

## 2. 도메인 모델 한 줄 요약

board의 현재 핵심 도메인 모델은 게시글이다.

게시글은 제목과 본문을 가지며,  
생성 시각과 수정 시각을 통해 기본 관리 정보를 가진다.

현재 단계에서는 게시글 CRUD를 검증할 수 있는 최소 도메인 모델을 우선한다.

---

## 3. 도메인 모델 정리 기준

현재 도메인 모델은 다음 기준을 따른다.

- 현재 단계에서는 게시글 도메인만 우선 정의한다.
- 댓글, 첨부파일, 사용자, 권한 모델은 현재 범위에 포함하지 않는다.
- Spring Boot 기반 layered 구조에서 구현 가능한 단순 모델을 우선한다.
- JPA Entity와 Domain 객체의 분리 여부는 현재 단계에서 확정하지 않는다.
- 도메인 모델은 DB 테이블 설계를 직접 확정하지 않는다.
- 도메인 모델은 API 요청/응답 DTO를 직접 확정하지 않는다.
- 미래 확장을 이유로 현재 도메인 모델을 과도하게 복잡하게 만들지 않는다.
- 현재 요구사항을 만족하는 최소 규칙을 명확히 하는 데 집중한다.

현재는 게시글 하나의 기본 CRUD 흐름에 필요한 도메인 개념만 다룬다.

## 6. 핵심 도메인 개념

현재 board의 핵심 도메인 개념은 다음과 같다.

- 게시글

게시글은 사용자가 작성하고 조회할 수 있는 게시판의 기본 단위다.

게시글은 현재 단계에서 다음 기능 흐름의 중심이 된다.

- 목록 조회
- 상세 조회
- 등록
- 수정
- 삭제

현재 단계에서 게시글은 단독 도메인으로 다룬다.  
사용자, 댓글, 첨부파일과의 관계는 아직 정의하지 않는다.

## 7. 게시글 모델

### 7.1 게시글 정의

게시글은 게시판에 등록되는 기본 콘텐츠 단위다.

게시글은 최소한 다음 정보를 가진다.

- 식별자
- 제목
- 본문
- 생성 시각
- 수정 시각

현재 단계에서 게시글은 제목과 본문을 중심으로 관리한다.

### 7.2 게시글 식별자

게시글 식별자는 게시글 하나를 구분하기 위한 값이다.

현재 단계의 기준은 다음과 같다.

- 게시글은 고유한 식별자를 가진다.
- 식별자는 목록, 상세, 수정, 삭제 흐름에서 사용된다.
- 존재하지 않는 식별자로 조회, 수정, 삭제할 수 없다.
- 식별자의 상세 타입은 DB 설계 문서에서 정한다.

현재 구현에서는 `Long` 타입 식별자를 우선 검토할 수 있다.  
다만 이 문서에서는 최종 DB 타입을 확정하지 않는다.

### 7.3 게시글 제목

게시글 제목은 게시글을 요약하는 필수 정보다.

현재 단계의 기준은 다음과 같다.

- 제목은 필수 값이다.
- 제목은 비어 있으면 안 된다.
- 제목은 목록과 상세 조회에서 확인할 수 있어야 한다.
- 제목은 등록 시 입력되어야 한다.
- 제목은 수정 시 변경될 수 있다.

제목의 최대 길이, 허용 문자, 공백 처리 기준은 이 문서에서 확정하지 않는다.  
필요 시 API 명세 또는 DB 설계 문서에서 구체화한다.

### 7.4 게시글 본문

게시글 본문은 게시글의 상세 내용을 담는 필수 정보다.

현재 단계의 기준은 다음과 같다.

- 본문은 필수 값이다.
- 본문은 비어 있으면 안 된다.
- 본문은 상세 조회에서 확인할 수 있어야 한다.
- 본문은 등록 시 입력되어야 한다.
- 본문은 수정 시 변경될 수 있다.

본문의 최대 길이, HTML 허용 여부, 줄바꿈 처리 기준은 이 문서에서 확정하지 않는다.  
필요 시 API 명세 또는 DB 설계 문서에서 구체화한다.

### 7.5 생성 시각

생성 시각은 게시글이 처음 생성된 시각이다.

현재 단계의 기준은 다음과 같다.

- 게시글 등록 시 생성 시각이 기록되어야 한다.
- 생성 시각은 게시글의 기본 관리 정보다.
- 생성 시각은 게시글 등록 이후 임의로 변경되지 않는 것을 원칙으로 한다.
- 생성 시각의 상세 저장 방식은 DB 설계 문서에서 정한다.

### 7.6 수정 시각

수정 시각은 게시글이 마지막으로 변경된 시각이다.

현재 단계의 기준은 다음과 같다.

- 게시글 등록 시 수정 시각도 초기화될 수 있다.
- 게시글 수정 시 수정 시각은 갱신되어야 한다.
- 수정 시각은 게시글의 기본 관리 정보다.
- 수정 시각의 상세 저장 방식은 DB 설계 문서에서 정한다.

## 8. 게시글 상태

현재 단계에서는 게시글 상태를 별도 상태값으로 세분화하지 않는다.

현재 기준은 다음과 같다.

- 생성된 게시글은 조회 가능 상태로 본다.
- 수정된 게시글은 수정된 내용으로 조회 가능해야 한다.
- 삭제된 게시글은 일반 조회 흐름에서 다시 조회되지 않아야 한다.
- 삭제 방식은 논리 삭제와 물리 삭제 중 아직 확정하지 않는다.

삭제 상태를 별도 필드로 둘지 여부는 DB 설계 문서에서 검토한다.

---

## 9. 게시글 생성 규칙

게시글 생성 규칙은 다음과 같다.

- 제목이 있어야 한다.
- 본문이 있어야 한다.
- 제목은 비어 있으면 안 된다.
- 본문은 비어 있으면 안 된다.
- 생성 시각이 기록되어야 한다.
- 수정 시각이 초기화되어야 한다.
- 생성된 게시글은 목록과 상세 조회에서 확인 가능해야 한다.

작성자 정보는 현재 단계에서 생성 규칙에 포함하지 않는다.  
로그인 사용자와 작성자 연계는 후속 auth 범위에서 검토한다.

---

## 10. 게시글 수정 규칙

게시글 수정 규칙은 다음과 같다.

- 수정 대상 게시글이 존재해야 한다.
- 제목을 수정할 수 있어야 한다.
- 본문을 수정할 수 있어야 한다.
- 수정 시 제목은 비어 있으면 안 된다.
- 수정 시 본문은 비어 있으면 안 된다.
- 수정 후 수정 시각이 갱신되어야 한다.
- 수정된 내용은 상세 조회에서 확인 가능해야 한다.

수정 권한은 현재 단계에서 정의하지 않는다.  
사용자 인증과 권한 연계는 후속 후보로 둔다.

---

## 11. 게시글 삭제 규칙

게시글 삭제 규칙은 다음과 같다.

- 삭제 대상 게시글이 존재해야 한다.
- 삭제 후 일반 조회 흐름에서 다시 조회되지 않아야 한다.
- 삭제 후 목록 조회 흐름으로 복귀할 수 있어야 한다.
- 존재하지 않는 게시글 삭제는 정상 삭제로 처리하지 않는다.

논리 삭제와 물리 삭제 중 어떤 방식을 사용할지는 현재 도메인 모델에서 확정하지 않는다.  
삭제 방식은 DB 설계 문서 또는 구현 계획 문서에서 필요 시 결정한다.

---

## 12. 게시글 조회 규칙

### 12.1 목록 조회

목록 조회 기준은 다음과 같다.

- 게시글 목록을 조회할 수 있어야 한다.
- 목록 항목은 게시글을 구분할 수 있어야 한다.
- 목록 항목에는 제목이 포함되어야 한다.
- 목록에서 상세 조회로 이어질 수 있어야 한다.
- 삭제된 게시글은 일반 목록 조회에 포함되지 않아야 한다.

목록 조회의 정렬, 페이징, 검색 조건은 현재 도메인 모델에서 확정하지 않는다.

### 12.2 상세 조회

상세 조회 기준은 다음과 같다.

- 게시글 식별자로 단건 게시글을 조회할 수 있어야 한다.
- 상세 조회 결과에는 제목이 포함되어야 한다.
- 상세 조회 결과에는 본문이 포함되어야 한다.
- 존재하지 않는 게시글은 정상 조회 결과로 처리하지 않는다.
- 삭제된 게시글은 일반 상세 조회에서 조회되지 않아야 한다.

오류 처리 방식은 API 명세 문서에서 구체화한다.

---

## 13. 입력 검증 기준

현재 도메인 모델에서 필요한 최소 입력 검증 기준은 다음과 같다.

- 제목은 비어 있으면 안 된다.
- 본문은 비어 있으면 안 된다.
- 게시글 식별자는 조회, 수정, 삭제 대상 게시글을 찾을 수 있어야 한다.
- 존재하지 않는 게시글은 정상 처리로 간주하지 않는다.

다만 다음 항목은 현재 문서에서 확정하지 않는다.

- 제목 최대 길이
- 본문 최대 길이
- 공백만 있는 문자열 허용 여부
- HTML 입력 허용 여부
- 금칙어 처리
- 특수문자 제한
- 상세 에러 메시지
- HTTP 상태 코드

위 항목은 API 명세 또는 DB 설계 문서에서 필요 시 구체화한다.

---

## 14. 도메인 모델과 Entity 관계

현재 단계에서는 도메인 모델과 JPA Entity의 분리 여부를 확정하지 않는다.

초기 layered 구조에서는 다음 방향을 우선 검토한다.

- 게시글 도메인 개념은 `Post` 또는 `PostEntity`로 구현할 수 있다.
- 단순 CRUD 검증 단계에서는 JPA Entity가 도메인 모델 역할을 일부 겸할 수 있다.
- 도메인 규칙이 복잡해지면 Domain 객체와 Entity 분리를 다시 검토한다.
- 현재 단계에서는 분리를 위한 분리보다 단순하고 검증 가능한 구조를 우선한다.

후속 문서별 역할은 다음과 같다.

- 애플리케이션 구조 문서는 Entity 위치와 패키지 구조를 정리한다.
- DB 설계 문서는 Entity와 테이블 매핑을 구체화한다.
- 구현 계획 문서는 실제 클래스 생성 순서를 정리한다.

---

## 15. 도메인 모델과 DTO 관계

DTO는 외부 요청/응답을 위한 객체다.

현재 기준은 다음과 같다.

- 등록 요청 DTO는 제목과 본문을 전달한다.
- 수정 요청 DTO는 제목과 본문을 전달한다.
- 목록 응답 DTO는 게시글 식별자와 제목 등 목록에 필요한 정보를 전달한다.
- 상세 응답 DTO는 제목과 본문 등 상세 조회에 필요한 정보를 전달한다.
- DTO 상세 필드는 API 명세 문서에서 구체화한다.
- Entity를 외부 API 응답으로 직접 노출하는 것은 피하는 방향을 우선 검토한다.

DTO 변환 위치는 애플리케이션 구조 문서의 Facade 기준을 따른다.

---

## 16. 도메인 모델과 Repository 관계

Repository는 게시글 도메인 데이터를 저장하고 조회하기 위한 데이터 접근 경계다.

현재 기준은 다음과 같다.

- Repository는 게시글 목록 조회를 지원해야 한다.
- Repository는 게시글 단건 조회를 지원해야 한다.
- Repository는 게시글 저장을 지원해야 한다.
- Repository는 게시글 수정 저장을 지원해야 한다.
- Repository는 게시글 삭제를 지원해야 한다.
- Repository는 게시글 식별자를 기준으로 게시글을 찾을 수 있어야 한다.

Repository 상세 메서드와 Spring Data JPA 사용 방식은 구현 계획 또는 DB 설계 문서에서 구체화한다.

---

## 17. 도메인 모델과 Service 관계

Service는 게시글 도메인 규칙을 적용하는 주요 위치다.

현재 기준은 다음과 같다.

- Service는 게시글 등록 규칙을 적용한다.
- Service는 게시글 수정 규칙을 적용한다.
- Service는 게시글 삭제 규칙을 적용한다.
- Service는 존재하지 않는 게시글에 대한 처리를 담당한다.
- Service는 트랜잭션 경계를 담당한다.
- Service는 Repository와 협력해 게시글 데이터를 저장하고 조회한다.

Facade는 Service를 호출하고 응답 조립을 담당할 수 있지만,  
게시글의 핵심 처리 규칙은 Service 중심으로 둔다.

---

## 18. 도메인 불변 조건

현재 단계에서 게시글이 지켜야 할 최소 불변 조건은 다음과 같다.

- 게시글은 식별 가능해야 한다.
- 제목은 비어 있으면 안 된다.
- 본문은 비어 있으면 안 된다.
- 생성된 게시글은 생성 시각을 가져야 한다.
- 생성된 게시글은 수정 시각을 가져야 한다.
- 수정된 게시글은 수정 시각이 갱신되어야 한다.
- 존재하지 않는 게시글은 정상 조회, 수정, 삭제 대상으로 처리되지 않아야 한다.
- 삭제된 게시글은 일반 조회 흐름에서 다시 조회되지 않아야 한다.

불변 조건을 Entity 생성자, 정적 팩토리, Service 검증, Bean Validation 중 어디에서 보장할지는 구현 단계에서 결정한다.

## 19. 이름 후보

현재 단계에서 사용할 수 있는 주요 모델 이름 후보는 다음과 같다.

| 개념 | 이름 후보 | 비고 |
|---|---|---|
| 게시글 | Post | 도메인 개념 이름 |
| 게시글 Entity | PostEntity | JPA Entity 이름 후보 |
| 게시글 Repository | PostRepository | 데이터 접근 객체 |
| 게시글 Service | PostService | 게시글 기능 처리 |
| 게시글 Facade | PostFacade | 흐름 조정 및 응답 조립 |
| 게시글 Controller | PostController | API 요청 진입점 |
| 등록 요청 | PostCreateRequest | API 요청 DTO |
| 수정 요청 | PostUpdateRequest | API 요청 DTO |
| 목록 응답 | PostListResponse | API 응답 DTO |
| 상세 응답 | PostDetailResponse | API 응답 DTO |

최종 클래스명은 애플리케이션 구조 문서, API 명세 문서, 구현 계획 문서에서 함께 정리한다.

## 20. 후속 확장 후보

현재 도메인 모델에는 포함하지 않지만, 이후 검토할 수 있는 확장 후보는 다음과 같다.

- 작성자
- 사용자
- 권한
- 댓글
- 첨부파일
- 카테고리
- 태그
- 좋아요 / 추천
- 조회수
- 검색 조건
- 게시글 상태
- 삭제 상태
- 공개 / 비공개
- 공지글
- 관리자 기능

이 항목들은 현재 단계의 확정 도메인 모델이 아니다.  
필요성이 확인되면 요구사항 문서 또는 별도 도메인 모델 문서에서 다룬다.

## 21. 도메인 결정 보류 사항

현재 도메인 모델에서는 다음 판단을 보류한다.

- `Post`와 `PostEntity` 분리 여부
- 게시글 식별자 타입 최종 확정
- 제목 최대 길이
- 본문 최대 길이
- 공백 문자열 처리 기준
- 논리 삭제와 물리 삭제 중 선택
- 삭제 상태 필드 도입 여부
- 생성/수정 시각 자동 관리 방식
- 작성자 필드 도입 여부
- 조회수 필드 도입 여부
- 게시글 상태값 도입 여부
- Bean Validation 적용 위치
- 도메인 생성자와 정적 팩토리 사용 여부
- 별도 mapper 도입 여부

위 항목은 현재 단계에서 성급히 확정하지 않는다.  
필요성이 검증되면 후속 문서에서 구체화한다.
