# specs/03-design.md

## 1. 문서 목적

이 문서는 board 프로젝트의 설계 문서다.

현재 단계에서 board 프로젝트의 요구사항을 어떤 구조와 책임으로 나눌지 정리한다.

이 문서는 상세 API 명세 문서가 아니다.  
또한 상세 DB 설계 문서나 상세 패키지 구조 문서도 아니다.

현재 문서의 목적은 다음과 같다.

- 요구사항 문서의 게시글 CRUD 흐름을 설계 관점으로 정리한다.
- Spring Boot 기반 layered 구조에서 주요 책임을 나눈다.
- Controller, Service, Repository의 기본 역할을 정리한다.
- 도메인 모델, API 명세, DB 설계 문서가 이어받을 기준을 제공한다.
- 구현 계획 문서가 작업 단위를 나눌 수 있도록 설계 기준을 제공한다.

---

## 2. 설계 한 줄 요약

board는 Spring Boot 기반 layered 구조로 게시글 CRUD 흐름을 구현한다.

현재 설계의 핵심은 게시글 기능을  
Controller, Service, Repository, Domain/Data Model 책임으로 나누고,  
후속 API/DB/구현 문서가 이어받을 기준을 만드는 것이다.

---

## 3. 설계 기준

현재 설계는 다음 기준을 따른다.

- 현재 단계에서는 layered architecture를 우선 적용한다.
- 헥사고날 구조를 전제로 설계하지 않는다.
- 게시글 CRUD 흐름을 중심으로 최소 구조를 정의한다.
- 상세 API, DB, 패키지 구조를 이 문서에서 과도하게 확정하지 않는다.
- 요구사항 문서에서 정의한 포함 범위와 제외 범위를 벗어나지 않는다.
- logging, auth, board 같은 재사용 후보 영역은 경계만 의식한다.
- 미래 확장을 이유로 현재 구조를 과도하게 복잡하게 만들지 않는다.
- 설계 문서는 후속 문서가 이어받을 구조적 기준을 제공하는 데 집중한다.

---

## 4. 현재 설계 범위

현재 설계에서 다루는 범위는 다음과 같다.

- 게시글 목록 조회 흐름
- 게시글 상세 조회 흐름
- 게시글 등록 흐름
- 게시글 수정 흐름
- 게시글 삭제 흐름
- 최소 입력 검증 위치
- 계층별 책임 분리
- 게시글 데이터의 기본 흐름
- logging 적용 위치 검토
- auth 연계 가능성 검토
- 후속 문서와의 연결 기준

현재 설계는 게시판 기본 CRUD 흐름을 첫 번째 설계 범위로 본다.

---

## 5. 현재 설계 제외 범위

현재 설계에서 직접 확정하지 않는 범위는 다음과 같다.

- 상세 API 엔드포인트 확정
- 상세 요청/응답 DTO 확정
- 상세 DB 테이블명, 컬럼명, 인덱스 확정
- 상세 패키지 구조 확정
- 댓글 구조 설계
- 첨부파일 구조 설계
- 사용자 회원가입 / 로그인 구조 설계
- 상세 권한 관리 설계
- 관리자 기능 설계
- 멀티테넌시 설계
- 대규모 검색, 통계, 분석 구조 설계
- auth와 board의 독립 배포 구조 확정
- logging starter 분리 구조 확정
- 헥사고날 구조 전환 설계

위 항목은 현재 단계에서 확정하지 않는다.  
필요성이 확인되면 후속 문서나 별도 후보 문서에서 다룬다.

---

## 6. 전체 구조 방향

현재 board는 Spring Boot 기반 layered 구조를 우선한다.

기본 계층 방향은 다음과 같다.

- Controller
- Service
- Repository
- Domain/Data Model
- Persistence

현재 단계에서는 단순한 layered 구조를 사용한다.

각 계층은 다음 흐름으로 협력한다.

    Client
      -> Controller
      -> Service
      -> Repository
      -> Persistence

Controller는 외부 요청을 받아 Service에 전달한다.  
Service는 게시글 기능의 주요 흐름을 조정한다.  
Repository는 데이터 저장소 접근을 담당한다.  
Persistence는 JPA 기반 저장 구조를 우선 검토한다.

---

## 7. 계층별 책임

### 7.1 Controller 책임

Controller는 외부 요청과 애플리케이션 내부 흐름의 진입점이다.

현재 단계에서 Controller의 책임은 다음과 같다.

- 게시글 목록 조회 요청을 받는다.
- 게시글 상세 조회 요청을 받는다.
- 게시글 등록 요청을 받는다.
- 게시글 수정 요청을 받는다.
- 게시글 삭제 요청을 받는다.
- 요청 값을 Service에 전달한다.
- Service 처리 결과를 응답 형태로 반환한다.
- 요청/응답의 상세 형식은 API 명세 문서로 넘긴다.

Controller는 비즈니스 흐름의 중심이 아니다.  
게시글 처리의 주요 판단은 Service에서 담당한다.

### 7.2 Service 책임

Service는 게시글 기능의 주요 애플리케이션 흐름을 담당한다.

현재 단계에서 Service의 책임은 다음과 같다.

- 게시글 목록 조회 흐름을 처리한다.
- 게시글 상세 조회 흐름을 처리한다.
- 게시글 등록 흐름을 처리한다.
- 게시글 수정 흐름을 처리한다.
- 게시글 삭제 흐름을 처리한다.
- 입력값의 기본 유효성을 확인한다.
- 존재하지 않는 게시글에 대한 처리를 판단한다.
- Repository를 통해 데이터 저장소와 협력한다.
- 트랜잭션 경계를 담당한다.

Service는 Controller나 Repository에 세부 책임을 과도하게 넘기지 않는다.

### 7.3 Repository 책임

Repository는 데이터 저장소 접근을 담당한다.

현재 단계에서 Repository의 책임은 다음과 같다.

- 게시글 목록 조회에 필요한 데이터 접근을 제공한다.
- 게시글 단건 조회에 필요한 데이터 접근을 제공한다.
- 게시글 등록에 필요한 저장 기능을 제공한다.
- 게시글 수정에 필요한 조회 및 저장 기능을 제공한다.
- 게시글 삭제에 필요한 조회 및 삭제 기능을 제공한다.
- JPA 기반 구현을 우선 검토한다.

Repository는 요청/응답 형식이나 화면 흐름을 알지 않는다.  
Repository는 데이터 접근 책임에 집중한다.

### 7.4 Domain/Data Model 책임

현재 단계에서 게시글은 최소 도메인 개념으로 다룬다.

게시글은 최소한 다음 정보를 가진다.

- 식별자
- 제목
- 본문
- 생성 시각
- 수정 시각

현재 설계에서는 도메인 모델과 JPA Entity의 분리 여부를 확정하지 않는다.  
초기 layered 구조에서는 단순하게 시작할 수 있다.

다만 후속 확장을 고려하여 다음 기준은 남긴다.

- 게시글의 핵심 개념은 도메인 모델 문서에서 정리한다.
- JPA Entity의 상세 구조는 DB 설계 문서에서 구체화한다.
- DTO, Entity, Domain 객체의 분리 여부는 애플리케이션 구조 문서에서 판단한다.

## 8. 게시글 CRUD 설계

### 8.1 게시글 목록 조회 설계

게시글 목록 조회 흐름은 다음과 같다.

    Client
      -> Controller
      -> Service
      -> Repository
      -> Persistence
      -> Repository
      -> Service
      -> Controller
      -> Client

현재 단계의 설계 기준은 다음과 같다.

- Controller는 목록 조회 요청을 받는다.
- Service는 목록 조회 흐름을 조정한다.
- Repository는 저장소에서 게시글 목록을 조회한다.
- 결과는 목록 응답에 필요한 형태로 반환된다.
- 목록 항목에는 게시글 식별 정보와 제목이 포함되어야 한다.
- 페이징, 정렬, 검색 조건은 현재 설계에서 상세 확정하지 않는다.

필요 시 API 명세 문서에서 목록 조회의 최소 요청/응답 기준을 정한다.

### 8.2 게시글 상세 조회 설계

게시글 상세 조회 흐름은 다음과 같다.

    Client
      -> Controller
      -> Service
      -> Repository
      -> Persistence

현재 단계의 설계 기준은 다음과 같다.

- Controller는 게시글 식별자를 받는다.
- Service는 게시글 조회 흐름을 처리한다.
- Repository는 식별자 기준으로 게시글을 조회한다.
- Service는 게시글이 존재하지 않는 경우를 처리한다.
- 상세 결과에는 제목과 본문이 포함되어야 한다.
- 오류 응답 형식은 API 명세 문서에서 구체화한다.

### 8.3 게시글 등록 설계

게시글 등록 흐름은 다음과 같다.

    Client
      -> Controller
      -> Service
      -> Repository
      -> Persistence

현재 단계의 설계 기준은 다음과 같다.

- Controller는 게시글 등록 요청을 받는다.
- Service는 제목과 본문 입력값을 검증한다.
- 제목은 비어 있으면 안 된다.
- 본문은 비어 있으면 안 된다.
- Service는 게시글 생성 흐름을 처리한다.
- Repository는 게시글을 저장한다.
- 등록 후 목록 또는 상세 흐름으로 이어질 수 있어야 한다.

작성자, 로그인 사용자, 권한 정보는 현재 단계에서 확정하지 않는다.

### 8.4 게시글 수정 설계

게시글 수정 흐름은 다음과 같다.

    Client
      -> Controller
      -> Service
      -> Repository
      -> Persistence

현재 단계의 설계 기준은 다음과 같다.

- Controller는 게시글 식별자와 수정 요청 값을 받는다.
- Service는 수정 대상 게시글을 조회한다.
- Service는 게시글이 존재하지 않는 경우를 처리한다.
- Service는 제목과 본문 입력값을 검증한다.
- 제목은 비어 있으면 안 된다.
- 본문은 비어 있으면 안 된다.
- Repository는 수정된 게시글을 저장한다.
- 수정 후 변경된 내용을 확인할 수 있어야 한다.

수정 권한과 사용자 인증 연계는 현재 설계에서 확정하지 않는다.

### 8.5 게시글 삭제 설계

게시글 삭제 흐름은 다음과 같다.

    Client
      -> Controller
      -> Service
      -> Repository
      -> Persistence

현재 단계의 설계 기준은 다음과 같다.

- Controller는 삭제 대상 게시글 식별자를 받는다.
- Service는 삭제 대상 게시글을 조회한다.
- Service는 게시글이 존재하지 않는 경우를 처리한다.
- Repository는 게시글 삭제를 수행한다.
- 삭제 후 목록 흐름으로 이어질 수 있어야 한다.
- 삭제된 게시글은 일반 조회 흐름에서 다시 조회되지 않아야 한다.

논리 삭제와 물리 삭제 중 어떤 방식을 사용할지는 이 문서에서 확정하지 않는다.  
DB 설계 문서 또는 구현 계획 문서에서 필요 시 결정한다.

## 9. 입력 검증 설계

현재 단계에서 입력 검증은 최소 기준만 둔다.

입력 검증 대상은 다음과 같다.

- 게시글 등록 시 제목
- 게시글 등록 시 본문
- 게시글 수정 시 제목
- 게시글 수정 시 본문
- 상세 조회, 수정, 삭제 시 게시글 식별자

현재 단계의 검증 기준은 다음과 같다.

- 제목은 비어 있으면 안 된다.
- 본문은 비어 있으면 안 된다.
- 게시글 식별자는 유효한 조회 기준이어야 한다.
- 존재하지 않는 게시글은 정상 결과로 처리하지 않는다.
- 잘못된 입력은 정상 등록 또는 정상 수정으로 처리하지 않는다.

검증 위치는 다음과 같이 나눈다.

- Controller는 요청 형식과 기본 바인딩을 담당한다.
- Service는 기능 흐름에서 필요한 유효성 판단을 담당한다.
- 상세 검증 방식과 에러 응답 형식은 API 명세 문서에서 구체화한다.

---

## 10. 오류 처리 설계

현재 단계에서 필요한 오류 처리 대상은 다음과 같다.

- 필수 입력값 누락
- 존재하지 않는 게시글 조회
- 존재하지 않는 게시글 수정
- 존재하지 않는 게시글 삭제
- 잘못된 게시글 식별자
- 저장 처리 실패

현재 문서에서는 오류 응답 형식을 확정하지 않는다.

다만 설계 기준은 다음과 같다.

- 오류 상황은 정상 성공 흐름과 구분되어야 한다.
- Service는 존재하지 않는 게시글을 정상 결과로 처리하지 않아야 한다.
- Controller는 오류 상황을 사용자 또는 클라이언트가 이해할 수 있는 응답으로 변환해야 한다.
- 공통 예외 처리 방식은 후속 설계 또는 API 명세 문서에서 구체화한다.

---

## 11. 트랜잭션 설계

현재 단계에서 트랜잭션은 Service 계층을 중심으로 관리한다.

트랜잭션이 필요한 흐름은 다음과 같다.

- 게시글 등록
- 게시글 수정
- 게시글 삭제

조회 흐름은 읽기 전용 트랜잭션 적용 가능성을 검토할 수 있다.

현재 설계 기준은 다음과 같다.

- Controller는 트랜잭션 경계를 갖지 않는다.
- Repository는 데이터 접근을 담당하되 전체 유스케이스 트랜잭션을 조정하지 않는다.
- Service가 기능 흐름 단위의 트랜잭션 경계를 담당한다.

---

## 12. 데이터 설계 연결 기준

현재 단계에서 게시글 데이터는 최소한 다음 항목을 가진다.

- 식별자
- 제목
- 본문
- 생성 시각
- 수정 시각

이 문서에서는 테이블명, 컬럼명, 제약 조건, 인덱스를 확정하지 않는다.

DB 설계 문서는 다음 내용을 이어받아야 한다.

- 게시글 기본 데이터 항목
- 제목과 본문의 필수 조건
- 생성 시각과 수정 시각 관리 기준
- 삭제 방식 결정 필요 여부
- JPA Entity 설계 기준
- Flyway 사용 여부 검토

---

## 13. API 설계 연결 기준

API 명세 문서는 이 설계 문서를 기준으로 다음 내용을 구체화한다.

- 게시글 목록 조회 API
- 게시글 상세 조회 API
- 게시글 등록 API
- 게시글 수정 API
- 게시글 삭제 API
- 요청 DTO
- 응답 DTO
- 오류 응답 형식
- 입력 검증 응답
- HTTP method와 URL
- 상태 코드 기준

현재 문서에서는 API 상세를 확정하지 않는다.  
API 명세 문서에서 별도로 정리한다.

애플리케이션 구조 문서는 이 설계 문서를 기준으로 다음 내용을 구체화한다.

- 실제 패키지 구조
- Controller 위치

- Repository 위치

실제 파일/패키지 배치는 애플리케이션 구조 문서에서 정리한다.

## 15. 공통 기능 및 재사용 후보 설계 기준

현재 board는 하나의 애플리케이션으로 시작한다.

다만 다음 영역은 경계를 의식한다.

- logging
- auth
- board

### 15.1 logging

logging은 공통 기능 후보로 본다.

현재 설계 기준은 다음과 같다.

- 요청 처리 흐름에서 필요한 최소 로그를 고려한다.
- Controller 또는 Service 흐름에서 추적 가능한 로그 위치를 검토한다.
- logging은 후속 공통 모듈 또는 starter 분리 가능성을 가진다.
- 현재 단계에서는 logging starter 구조를 확정하지 않는다.

### 15.2 auth

auth는 현재 단계에서 상세 구현 범위를 확정하지 않는다.

현재 설계 기준은 다음과 같다.

- 게시글 CRUD는 우선 인증 없이 최소 흐름을 검증할 수 있다.
- 작성자, 권한, 로그인 사용자 연계는 후속 후보로 둔다.
- auth와 board의 경계는 장기적으로 고려한다.
- 현재 단계에서는 auth 독립 배포 구조를 확정하지 않는다.

### 15.3 board

board는 현재 애플리케이션의 핵심 기능 영역이다.

현재 설계 기준은 다음과 같다.

- 게시글 CRUD는 board의 핵심 기능으로 본다.
- board 기능은 다른 공통 기능과 과도하게 결합하지 않는다.
- auth, logging과의 연결 지점은 후속 구조 문서에서 검토할 수 있도록 남긴다.

## 17. 현재 설계의 의도적 보류 사항

현재 설계에서는 다음 판단을 보류한다.

- DTO와 Entity의 분리 수준
- Domain 객체와 JPA Entity의 분리 여부
- 논리 삭제와 물리 삭제 중 선택
- 페이징 기본 적용 여부
- 공통 응답 포맷 적용 여부
- 전역 예외 처리 방식
- Bean Validation 적용 방식
- Flyway 즉시 적용 여부
- logging 공통 모듈화 방식
- auth 연계 방식
- 멀티모듈 전환 여부
- 헥사고날 구조 전환 여부

이 항목들은 현재 설계에서 성급히 확정하지 않는다.  
필요성이 검증되면 후속 문서에서 구체화한다.
