# specs/03-design.md

## 1. 문서 목적

이 문서는 board 프로젝트의 설계 문서다.

현재 단계에서 board 프로젝트의 요구사항을 Spring Boot 기반 layered 구조로 어떻게 나눌지 정리한다.  
이 문서는 상세 API 명세, 상세 DB 설계, 상세 패키지 구조, 구현 계획을 확정하지 않는다.

목적은 다음과 같다.

- 게시글 CRUD 요구사항을 설계 관점으로 정리한다.
- Controller, Service, Repository, Domain/Data Model의 책임을 나눈다.
- 입력 검증, 오류 처리, 트랜잭션 경계의 설계 기준을 둔다.
- 도메인 모델, API 명세, DB 설계, 애플리케이션 구조, 구현 계획 문서가 이어받을 기준을 제공한다.

---

## 2. 설계 한 줄 요약

board는 Spring Boot 기반 layered 구조로 게시글 CRUD 흐름을 구현한다.

현재 설계의 핵심은 게시글 기능을 Controller, Service, Repository, Domain/Data Model 책임으로 나누고, 후속 문서가 상세 결정을 이어받을 기준을 만드는 것이다.

---

## 3. 현재 설계 결정

현재 확정한 설계 결정은 다음과 같다.

- board는 우선 단순 layered architecture를 따른다.
- 헥사고날 구조는 현재 설계 기준으로 채택하지 않는다.
- 게시글 CRUD를 첫 설계 범위로 둔다.
- Controller는 요청 진입과 응답 반환을 담당한다.
- Service는 게시글 기능 흐름, 유효성 판단, 트랜잭션 경계를 담당한다.
- Repository는 데이터 저장소 접근을 담당한다.
- Persistence는 JPA 기반 저장 구조를 우선 검토한다.
- 상세 API, DB, 패키지 배치, 구현 순서는 후속 문서에서 정한다.

---

## 4. 현재 설계 범위

현재 설계 범위는 다음과 같다.

- 게시글 목록 조회
- 게시글 상세 조회
- 게시글 등록
- 게시글 수정
- 게시글 삭제
- 최소 입력 검증 위치
- 계층별 책임 분리
- 게시글 데이터의 기본 흐름
- 오류 처리 기준
- 트랜잭션 경계
- logging, auth 연계 가능성의 경계 인식
- 후속 문서와의 연결 기준

---

## 5. 현재 설계 제외 범위

이 문서에서 확정하지 않는 범위는 다음과 같다.

- 상세 API 엔드포인트, HTTP method, 상태 코드
- 상세 요청/응답 DTO
- 상세 DB 테이블명, 컬럼명, 인덱스
- 상세 패키지 구조
- 댓글, 첨부파일, 관리자 기능
- 회원가입, 로그인, 상세 권한 관리
- 멀티테넌시, 검색, 통계, 분석
- auth와 board의 독립 배포 구조
- logging starter 분리 구조
- 헥사고날 구조 전환

위 항목은 필요성이 확인되면 후속 문서에서 다룬다.

---

## 6. 전체 구조 방향

기본 흐름은 다음과 같다.

    Client
      -> Controller
      -> Service
      -> Repository
      -> Persistence

계층 책임은 다음 기준을 따른다.

- Controller: 외부 요청 수신, Service 호출, 응답 반환
- Service: 유스케이스 흐름 조정, 유효성 판단, 예외 판단, 트랜잭션 경계
- Repository: 게시글 데이터 접근
- Domain/Data Model: 게시글 핵심 데이터와 상태 표현
- Persistence: JPA 기반 저장 구조 후보

---

## 7. 계층별 책임

### 7.1 Controller

Controller는 외부 요청과 내부 흐름의 진입점이다.

- 게시글 CRUD 요청을 받는다.
- 요청 값을 Service에 전달한다.
- Service 처리 결과를 응답 형태로 반환한다.
- 요청/응답 상세 형식은 API 명세 문서로 넘긴다.
- 비즈니스 판단을 직접 담당하지 않는다.

### 7.2 Service

Service는 게시글 기능의 애플리케이션 흐름을 담당한다.

- 목록, 상세, 등록, 수정, 삭제 흐름을 처리한다.
- 제목, 본문, 식별자 등 기본 유효성을 판단한다.
- 존재하지 않는 게시글 상황을 정상 흐름과 구분한다.
- Repository와 협력한다.
- 등록, 수정, 삭제의 트랜잭션 경계를 담당한다.

### 7.3 Repository

Repository는 데이터 저장소 접근을 담당한다.

- 게시글 목록 조회
- 게시글 단건 조회
- 게시글 저장
- 게시글 수정 저장
- 게시글 삭제
- JPA 기반 구현 우선 검토

Repository는 요청/응답 형식이나 화면 흐름을 알지 않는다.

### 7.4 Domain/Data Model

현재 게시글은 최소 도메인 개념으로 다룬다.

최소 데이터는 다음과 같다.

- 식별자
- 제목
- 본문
- 생성 시각
- 수정 시각

도메인 모델과 JPA Entity의 분리 여부는 현재 문서에서 확정하지 않는다.  
도메인 모델, DB 설계, 애플리케이션 구조 문서에서 이어서 판단한다.

## 8. 게시글 CRUD 설계 기준

### 8.1 목록 조회

- Controller는 목록 조회 요청을 받는다.
- Service는 목록 조회 흐름을 조정한다.
- Repository는 게시글 목록을 조회한다.
- 목록 항목에는 최소한 식별 정보와 제목이 포함되어야 한다.
- 페이징, 정렬, 검색 조건은 후속 문서에서 결정한다.

### 8.2 상세 조회

- Controller는 게시글 식별자를 받는다.
- Service는 조회 흐름을 처리한다.
- Repository는 식별자 기준으로 게시글을 조회한다.
- 존재하지 않는 게시글은 정상 결과로 처리하지 않는다.
- 상세 결과에는 제목과 본문이 포함되어야 한다.

### 8.3 등록

- Controller는 등록 요청을 받는다.
- Service는 제목과 본문을 검증한다.
- 제목과 본문은 비어 있으면 안 된다.
- Service는 게시글 생성 흐름을 처리한다.
- Repository는 게시글을 저장한다.
- 작성자, 로그인 사용자, 권한 정보는 현재 확정하지 않는다.

### 8.4 수정

- Controller는 식별자와 수정 요청 값을 받는다.
- Service는 수정 대상 게시글을 조회한다.
- 존재하지 않는 게시글은 정상 수정으로 처리하지 않는다.
- 제목과 본문은 비어 있으면 안 된다.
- Repository는 수정된 게시글을 저장한다.
- 수정 권한과 인증 연계는 현재 확정하지 않는다.

### 8.5 삭제

- Controller는 삭제 대상 식별자를 받는다.
- Service는 삭제 대상 게시글을 조회한다.
- 존재하지 않는 게시글은 정상 삭제로 처리하지 않는다.
- Repository는 삭제를 수행한다.
- 삭제된 게시글은 일반 조회 흐름에서 다시 조회되지 않아야 한다.
- 논리 삭제와 물리 삭제는 후속 문서에서 결정한다.

---

## 9. 입력 검증 설계

현재 입력 검증은 최소 기준만 둔다.

- 제목은 비어 있으면 안 된다.
- 본문은 비어 있으면 안 된다.
- 게시글 식별자는 유효한 조회 기준이어야 한다.
- 존재하지 않는 게시글은 정상 결과로 처리하지 않는다.
- 잘못된 입력은 정상 등록 또는 정상 수정으로 처리하지 않는다.

검증 위치는 다음과 같이 나눈다.

- Controller: 요청 형식과 기본 바인딩
- Service: 기능 흐름에 필요한 유효성 판단
- API 명세 문서: 상세 검증 방식과 오류 응답 형식

---

## 10. 오류 처리 설계

현재 필요한 오류 처리 대상은 다음과 같다.

- 필수 입력값 누락
- 존재하지 않는 게시글 조회
- 존재하지 않는 게시글 수정
- 존재하지 않는 게시글 삭제
- 잘못된 게시글 식별자
- 저장 처리 실패

설계 기준은 다음과 같다.

- 오류 상황은 정상 성공 흐름과 구분한다.
- Service는 존재하지 않는 게시글을 정상 결과로 처리하지 않는다.
- Controller는 오류를 클라이언트가 이해할 수 있는 응답으로 변환한다.
- 공통 예외 처리 방식과 오류 응답 형식은 후속 문서에서 정한다.

---

## 11. 트랜잭션 설계

트랜잭션은 Service 계층을 중심으로 관리한다.

트랜잭션이 필요한 흐름은 다음과 같다.

- 게시글 등록
- 게시글 수정
- 게시글 삭제

조회 흐름은 읽기 전용 트랜잭션 적용 가능성을 후속 문서에서 검토할 수 있다.

## 13. 후속 후보

현재 확정하지 않고 후속 후보로 남기는 항목은 다음과 같다.

- auth 연계 방식
- 멀티모듈 전환
