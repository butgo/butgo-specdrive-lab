# specs/06-api-spec.md

## 1. 문서 목적

이 문서는 board 프로젝트의 API 명세 문서다.

현재 단계에서 게시글 CRUD 기능을 외부에서 어떤 HTTP API로 사용할 수 있는지 정리한다.

이 문서는 상세 DB 설계 문서가 아니다.  
또한 구현 계획 문서나 테스트 계획 문서도 아니다.

현재 문서의 목적은 다음과 같다.

- 게시글 CRUD 기능의 API 목록을 정의한다.
- 요청 URL, HTTP Method, 요청 DTO, 응답 DTO의 기본 방향을 정리한다.
- 입력 검증과 오류 응답의 최소 기준을 정리한다.
- Controller, Facade, Service 구현이 따라야 할 API 기준을 제공한다.
- 구현 계획 문서가 API 구현 작업을 나눌 수 있도록 기준을 제공한다.

---

## 2. API 명세 한 줄 요약

board의 현재 API는 게시글 CRUD를 위한 최소 HTTP API다.

현재 단계에서는 다음 흐름을 우선한다.

- 게시글 목록 조회
- 게시글 상세 조회
- 게시글 등록
- 게시글 수정
- 게시글 삭제

API는 Spring Boot 기반 Controller에서 진입하고,  
Controller는 Facade를 통해 게시글 기능 흐름을 위임한다.

---

## 3. API 정리 기준

현재 API 명세는 다음 기준을 따른다.

- 현재 단계에서는 게시글 CRUD API만 정의한다.
- 인증과 권한이 필요한 API 구조는 현재 단계에서 확정하지 않는다.
- 댓글, 첨부파일, 좋아요, 관리자 API는 현재 범위에 포함하지 않는다.
- API는 HTTP 기반 REST 스타일을 우선한다.
- API 요청과 응답에는 DTO를 사용한다.
- Entity를 외부 API 응답으로 직접 노출하지 않는다.
- 상세 공통 응답 포맷은 현재 단계에서 확정하지 않는다.
- API는 구현과 검증이 가능한 최소 수준으로 정의한다.
- 미래 확장을 이유로 현재 API를 과도하게 복잡하게 만들지 않는다.

## 4. 현재 API 범위

현재 API 명세에서 다루는 범위는 다음과 같다.

- 게시글 목록 조회 API
- 게시글 상세 조회 API
- 게시글 등록 API
- 게시글 수정 API
- 게시글 삭제 API
- 게시글 등록 요청 DTO
- 게시글 수정 요청 DTO

- 기본 오류 응답 기준
- 기본 입력 검증 기준

현재는 게시글 하나의 기본 CRUD 흐름에 필요한 API만 다룬다.

## 5. 현재 API 제외 범위

현재 API 명세에서 직접 정의하지 않는 범위는 다음과 같다.

- 사용자 회원가입 API
- 사용자 로그인 API
- 인증 토큰 API
- 권한 관리 API
- 댓글 API
- 대댓글 API
- 첨부파일 API
- 좋아요 / 추천 API
- 검색 고도화 API
- 통계 API
- 관리자 API
- 알림 API
- 멀티테넌시 API
- 모바일 앱 전용 API
- 외부 연동 API

위 항목들은 현재 단계에서 정의하지 않는다.  
필요성이 검증되면 후속 요구사항 후보 또는 별도 API 문서에서 다룬다.

## 6. API 기본 경로

현재 게시글 API의 기본 경로는 다음과 같이 둔다.

    /api/posts

게시글 단건 API는 게시글 식별자를 경로 변수로 사용한다.

    /api/posts/{postId}

현재 단계에서는 API 버전 경로를 확정하지 않는다.

예를 들어 `/api/v1/posts` 같은 버전 경로는 후속 확장 시 검토할 수 있다.

---

## 7. API 목록

현재 단계에서 제공할 API 목록은 다음과 같다.

| 기능 | Method | URL | 설명 |
|---|---:|---|---|
| 게시글 목록 조회 | GET | `/api/posts` | 게시글 목록을 조회한다. |
| 게시글 상세 조회 | GET | `/api/posts/{postId}` | 게시글 하나를 조회한다. |
| 게시글 등록 | POST | `/api/posts` | 새 게시글을 등록한다. |
| 게시글 수정 | PUT | `/api/posts/{postId}` | 기존 게시글을 수정한다. |
| 게시글 삭제 | DELETE | `/api/posts/{postId}` | 기존 게시글을 삭제한다. |

현재 단계에서는 `PUT`을 게시글 전체 수정 기준으로 사용한다.  
부분 수정이 필요하면 후속 단계에서 `PATCH` 도입을 검토한다.

## 8. 게시글 목록 조회 API

### 8.1 기본 정보

| 항목 | 값 |
|---|---|
| 기능 | 게시글 목록 조회 |
| Method | `GET` |
| URL | `/api/posts` |
| Request Body | 없음 |
| Response Body | 게시글 목록 응답 |

### 8.2 요청

현재 단계에서는 목록 조회 요청에 별도 Query Parameter를 확정하지 않는다.

페이징, 정렬, 검색 조건은 현재 단계에서 보류한다.  
필요성이 확인되면 후속 API 명세에서 추가한다.

### 8.3 응답

응답은 게시글 목록을 반환한다.

현재 단계에서 목록 항목은 최소한 다음 정보를 포함한다.

- 게시글 식별자
- 제목
- 생성 시각
- 수정 시각

예상 응답 DTO는 다음과 같다.

    PostListResponse

예상 응답 예시는 다음과 같다.

    [
      {
        "id": 1,
        "title": "첫 번째 게시글",
        "createdAt": "2026-01-01T10:00:00",
        "updatedAt": "2026-01-01T10:00:00"
      }
    ]

### 8.4 성공 상태 코드

| 상황 | HTTP Status |
|---|---:|
| 목록 조회 성공 | `200 OK` |

## 10. 게시글 등록 API

### 10.1 기본 정보

| 항목 | 값 |
|---|---|
| 기능 | 게시글 등록 |
| Method | `POST` |
| URL | `/api/posts` |
| Request Body | 게시글 등록 요청 |
| Response Body | 게시글 상세 응답 또는 생성 결과 응답 |

### 10.2 요청

예상 요청 DTO는 다음과 같다.

    PostCreateRequest

요청 필드는 다음과 같다.

| 필드 | 타입 후보 | 필수 | 설명 |
|---|---|---|---|
| title | String | 예 | 게시글 제목 |
| content | String | 예 | 게시글 본문 |

예상 요청 예시는 다음과 같다.

    {
      "title": "첫 번째 게시글",
      "content": "게시글 본문입니다."
    }

### 10.3 요청 검증

현재 단계에서 등록 요청은 다음 검증을 만족해야 한다.

- 제목은 비어 있으면 안 된다.
- 본문은 비어 있으면 안 된다.

제목과 본문의 최대 길이는 현재 단계에서 확정하지 않는다.  
필요 시 API 명세 또는 DB 설계 문서에서 후속으로 정한다.

### 10.4 응답

등록 성공 시 생성된 게시글 정보를 반환할 수 있다.

예상 응답 DTO는 다음과 같다.

    PostDetailResponse

예상 응답 예시는 다음과 같다.

    {
      "id": 1,
      "title": "첫 번째 게시글",
      "content": "게시글 본문입니다.",
      "createdAt": "2026-01-01T10:00:00",
      "updatedAt": "2026-01-01T10:00:00"
    }

### 10.5 성공 상태 코드

| 상황 | HTTP Status |
|---|---:|
| 게시글 등록 성공 | `201 Created` |

### 10.6 오류 상태 코드

| 상황 | HTTP Status |
|---|---:|
| 필수 입력값 누락 | `400 Bad Request` |
| 요청 형식 오류 | `400 Bad Request` |

## 14. 오류 응답 기준

현재 단계에서는 공통 오류 응답 구조를 확정하지 않는다.

다만 오류 응답은 최소한 다음 정보를 표현할 수 있어야 한다.

- 오류 코드 또는 오류 유형
- 오류 메시지
- 요청 경로
- 발생 시각

예상 오류 응답 예시는 다음과 같다.

    {
      "code": "POST_NOT_FOUND",
      "message": "게시글을 찾을 수 없습니다.",
      "path": "/api/posts/1",
      "timestamp": "2026-01-01T10:00:00"
    }

최종 오류 응답 구조는 공통 응답 구조 도입 여부와 함께 결정한다.

---

## 15. 기본 오류 코드 후보

현재 단계에서 사용할 수 있는 오류 코드 후보는 다음과 같다.

| 코드 | 설명 | HTTP Status |
|---|---|---:|
| INVALID_REQUEST | 잘못된 요청 | `400 Bad Request` |
| VALIDATION_ERROR | 입력 검증 실패 | `400 Bad Request` |
| POST_NOT_FOUND | 게시글을 찾을 수 없음 | `404 Not Found` |
| INTERNAL_ERROR | 서버 내부 오류 | `500 Internal Server Error` |

현재 오류 코드는 후보이며, 구현 단계에서 최소 범위로 조정할 수 있다.

---

## 16. 입력 검증 기준

현재 API 입력 검증 기준은 다음과 같다.

- 게시글 등록 시 `title`은 비어 있으면 안 된다.
- 게시글 등록 시 `content`는 비어 있으면 안 된다.
- 게시글 수정 시 `title`은 비어 있으면 안 된다.
- 게시글 수정 시 `content`는 비어 있으면 안 된다.
- 상세 조회, 수정, 삭제 시 `postId`는 유효한 식별자여야 한다.
- 존재하지 않는 `postId`는 정상 처리하지 않는다.

현재 문서에서는 다음 기준을 확정하지 않는다.

- 제목 최대 길이
- 본문 최대 길이
- 공백만 있는 문자열 허용 여부
- HTML 입력 허용 여부
- 상세 검증 메시지
- 필드별 오류 응답 구조

필요 시 API 명세 후속 보강 또는 구현 계획에서 구체화한다.

---

## 17. 인증과 권한 기준

현재 단계의 게시글 CRUD API는 인증 없이 검증 가능해야 한다.

현재 기준은 다음과 같다.

- 사용자 로그인은 현재 API 범위에 포함하지 않는다.
- 작성자 기반 권한 검사는 현재 API 범위에 포함하지 않는다.
- 게시글 등록, 수정, 삭제 권한은 현재 단계에서 확정하지 않는다.
- auth 연계는 후속 요구사항 후보로 둔다.

향후 auth가 도입되면 다음 항목을 다시 검토한다.

- 작성자 식별 방식
- 게시글 수정 권한
- 게시글 삭제 권한
- 인증 실패 응답
- 권한 부족 응답
- 사용자별 게시글 조회 기준

---

## 18. 페이징, 정렬, 검색 기준

현재 단계에서는 페이징, 정렬, 검색 조건을 API 필수 범위로 확정하지 않는다.

다만 후속 확장을 고려해 다음 항목은 후보로 둔다.

- 목록 조회 페이징
- 최신순 정렬
- 제목 검색
- 본문 검색
- 작성자 검색
- 상태 필터

현재 API는 기본 CRUD 흐름 검증을 우선한다.  
페이징, 정렬, 검색은 필요성이 확인되면 후속 API 명세에서 다룬다.

---

## 19. Controller 연결 기준

Controller는 이 API 명세를 기준으로 외부 요청을 받는다.

현재 기준은 다음과 같다.

- `PostController`는 `/api/posts` 기본 경로를 가진다.
- Controller는 요청 DTO를 받는다.
- Controller는 경로 변수 `postId`를 받는다.
- Controller는 Facade를 호출한다.
- Controller는 Facade 결과를 API 응답으로 반환한다.
- Controller는 비즈니스 로직을 직접 처리하지 않는다.

예상 Controller 이름은 다음과 같다.

    PostController

---

## 20. Facade 연결 기준

Facade는 Controller와 Service 사이에서 API 흐름을 조정한다.

현재 기준은 다음과 같다.

- `PostFacade`는 Controller 요청을 받아 Service를 호출한다.
- Facade는 응답 DTO를 조립할 수 있다.
- Facade는 Entity를 외부 응답으로 직접 노출하지 않도록 경계를 제공한다.
- Facade는 Repository를 직접 호출하지 않는다.
- Facade는 핵심 업무 검증을 과도하게 담당하지 않는다.

예상 Facade 이름은 다음과 같다.

    PostFacade

---

## 21. Service 연결 기준

Service는 API 요청으로부터 시작된 게시글 기능의 핵심 처리를 담당한다.

현재 기준은 다음과 같다.

- `PostService`는 게시글 목록 조회를 처리한다.
- `PostService`는 게시글 상세 조회를 처리한다.
- `PostService`는 게시글 등록을 처리한다.
- `PostService`는 게시글 수정을 처리한다.
- `PostService`는 게시글 삭제를 처리한다.
- Service는 존재하지 않는 게시글을 정상 처리하지 않는다.
- Service는 트랜잭션 경계를 담당한다.

예상 Service 이름은 다음과 같다.

    PostService

---

## 22. API 결정 보류 사항

현재 API 명세에서는 다음 판단을 보류한다.

- API 버전 경로 도입 여부
- 공통 응답 포맷 적용 여부
- 오류 응답 상세 구조
- 필드별 검증 오류 응답 구조
- 제목 최대 길이
- 본문 최대 길이
- 공백 문자열 처리 기준
- 페이징 기본 적용 여부
- 정렬 기본 기준
- 검색 API 도입 여부
- 인증 API 연계 여부
- 작성자 필드 응답 포함 여부
- 삭제 성공 응답 본문 포함 여부
- `PUT`과 `PATCH` 병행 여부
- Swagger/OpenAPI 문서화 방식

위 항목들은 현재 단계에서 성급히 확정하지 않는다.  
필요성이 검증되면 후속 문서에서 구체화한다.
