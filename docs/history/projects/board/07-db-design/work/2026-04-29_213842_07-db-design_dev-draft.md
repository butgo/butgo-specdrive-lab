# specs/07-db-design.md

## 1. 문서 목적

이 문서는 board 프로젝트의 DB 설계 문서다.

현재 단계에서 게시글 CRUD 기능을 저장하기 위한 최소 데이터 구조를 정리한다.

이 문서는 상세 구현 계획 문서가 아니다.  
또한 API 명세 문서나 애플리케이션 구조 문서도 아니다.

현재 문서의 목적은 다음과 같다.

- 게시글 도메인 모델을 DB 구조로 옮기기 위한 기준을 정리한다.
- 게시글 Entity와 테이블의 기본 구조를 정의한다.
- 게시글 CRUD 구현에 필요한 최소 컬럼과 제약 조건을 정리한다.
- JPA 기반 persistence와 Flyway 적용 가능성을 검토한다.
- 구현 계획 문서가 DB 관련 작업 단위를 나눌 수 있도록 기준을 제공한다.

---

## 2. DB 설계 한 줄 요약

board의 현재 DB 설계는 게시글 CRUD를 위한 최소 테이블을 정의한다.

현재 단계의 핵심은 `posts` 테이블을 기준으로  
게시글의 식별자, 제목, 본문, 생성 시각, 수정 시각을 저장할 수 있게 하는 것이다.

---

## 3. DB 설계 기준

현재 DB 설계는 다음 기준을 따른다.

- 현재 단계에서는 게시글 테이블만 우선 정의한다.
- 사용자, 댓글, 첨부파일, 권한 테이블은 현재 범위에 포함하지 않는다.
- Spring Boot + JPA 기반 persistence 적용을 우선 검토한다.
- DB 구조는 현재 게시글 CRUD 요구사항을 만족하는 최소 범위로 정의한다.
- 과도한 정규화나 공통 테이블 일반화는 현재 단계에서 하지 않는다.
- Flyway 사용 가능성을 고려하되, 적용 여부는 구현 단계에서 최종 결정한다.
- 논리 삭제와 물리 삭제 중 하나를 성급히 확정하지 않는다.
- 현재 단계에서는 단순하고 검증 가능한 DB 구조를 우선한다.

---

## 4. 현재 DB 범위

현재 DB 설계 문서에서 다루는 범위는 다음과 같다.

- 게시글 테이블
- 게시글 식별자 컬럼
- 게시글 제목 컬럼
- 게시글 본문 컬럼
- 게시글 생성 시각 컬럼
- 게시글 수정 시각 컬럼
- 기본 제약 조건
- JPA Entity 매핑 방향
- Flyway migration 파일 위치
- 삭제 방식 검토 기준
- 후속 확장 후보

현재는 게시글 하나의 기본 CRUD 흐름에 필요한 DB 구조만 다룬다.

## 5. 현재 DB 제외 범위

현재 DB 설계에서 직접 정의하지 않는 범위는 다음과 같다.

- 사용자 테이블
- 작성자 테이블
- 인증 관련 테이블
- 권한 관련 테이블
- 댓글 테이블
- 대댓글 테이블
- 첨부파일 테이블
- 좋아요 / 추천 테이블
- 게시판 카테고리 테이블
- 태그 테이블
- 검색 인덱스 고도화
- 통계 테이블
- 알림 테이블
- 관리자 기능 테이블
- 멀티테넌시 테이블
- 공통 코드 테이블
- 이력 테이블

위 항목들은 현재 단계에서 정의하지 않는다.  
필요성이 검증되면 후속 요구사항 후보 또는 별도 DB 설계 문서에서 다룬다.

## 6. 테이블 목록

현재 단계에서 우선 정의하는 테이블은 다음과 같다.

| 테이블명 | 설명 | 현재 범위 |
|---|---|---|
| posts | 게시글 기본 정보 저장 | 포함 |

현재 단계에서는 `posts` 테이블 하나로 게시글 CRUD를 검증한다.

## 7. posts 테이블 개요

`posts` 테이블은 게시글의 기본 정보를 저장한다.

현재 단계에서 `posts` 테이블은 다음 정보를 가진다.

- 게시글 식별자
- 제목
- 본문
- 생성 시각
- 수정 시각

현재 단계에서는 작성자, 조회수, 상태, 삭제 여부 컬럼을 기본 포함하지 않는다.  
필요성이 확인되면 후속 단계에서 추가한다.

---

## 8. posts 테이블 컬럼

현재 `posts` 테이블의 컬럼 후보는 다음과 같다.

| 컬럼명 | 타입 후보 | 필수 | 설명 |
|---|---|---:|---|
| id | BIGINT | 예 | 게시글 식별자 |
| title | VARCHAR(255) | 예 | 게시글 제목 |
| content | TEXT | 예 | 게시글 본문 |
| created_at | TIMESTAMP | 예 | 생성 시각 |
| updated_at | TIMESTAMP | 예 | 수정 시각 |

현재 컬럼 타입은 후보이며, 실제 DB 제품과 JPA 매핑 방식에 따라 조정할 수 있다.

---

## 9. 컬럼별 설계 기준

### 9.1 id

`id`는 게시글을 식별하는 기본 키다.

현재 기준은 다음과 같다.

- 게시글마다 고유해야 한다.
- 목록, 상세, 수정, 삭제 API에서 게시글 식별자로 사용한다.
- JPA Entity에서는 `Long` 타입을 우선 검토한다.
- DB에서는 `BIGINT` 타입을 우선 검토한다.
- 자동 증가 전략 사용을 우선 검토한다.

ID 생성 전략은 실제 DB와 JPA 설정에 맞춰 구현 단계에서 결정한다.

### 9.2 title

`title`은 게시글 제목이다.

현재 기준은 다음과 같다.

- 필수 값이다.
- 비어 있으면 안 된다.
- 목록 조회와 상세 조회에 포함된다.
- 등록과 수정 시 입력된다.
- DB에서는 `VARCHAR(255)` 타입을 우선 검토한다.

제목 최대 길이는 현재 255자를 후보로 둔다.  
최종 길이는 API 검증 기준과 함께 조정할 수 있다.

### 9.3 content

`content`는 게시글 본문이다.

현재 기준은 다음과 같다.

- 필수 값이다.
- 비어 있으면 안 된다.
- 상세 조회에 포함된다.
- 등록과 수정 시 입력된다.
- DB에서는 `TEXT` 타입을 우선 검토한다.

본문 최대 길이, HTML 허용 여부, 줄바꿈 처리 기준은 현재 단계에서 확정하지 않는다.

### 9.4 created_at

`created_at`은 게시글 생성 시각이다.

현재 기준은 다음과 같다.

- 필수 값이다.
- 게시글 등록 시 기록된다.
- 게시글 생성 이후 임의로 변경하지 않는 것을 원칙으로 한다.
- JPA auditing 또는 애플리케이션 코드에서 자동 설정할 수 있다.

생성 시각 자동 관리 방식은 구현 단계에서 결정한다.

### 9.5 updated_at

`updated_at`은 게시글 수정 시각이다.

현재 기준은 다음과 같다.

- 필수 값이다.
- 게시글 등록 시 초기화될 수 있다.
- 게시글 수정 시 갱신되어야 한다.
- JPA auditing 또는 애플리케이션 코드에서 자동 설정할 수 있다.

수정 시각 자동 관리 방식은 구현 단계에서 결정한다.

---

## 10. 기본 키와 제약 조건

현재 단계에서 `posts` 테이블의 기본 제약 조건은 다음과 같다.

| 제약 | 대상 | 설명 |
|---|---|---|
| PRIMARY KEY | id | 게시글 식별자 |
| NOT NULL | title | 제목 필수 |
| NOT NULL | content | 본문 필수 |
| NOT NULL | created_at | 생성 시각 필수 |
| NOT NULL | updated_at | 수정 시각 필수 |

현재 단계에서는 unique 제약을 추가하지 않는다.

제목 중복은 허용한다.  
게시글 제목은 식별자가 아니므로 중복될 수 있다.

---

## 11. 인덱스 기준

현재 단계에서는 기본 키 인덱스 외에 별도 인덱스를 필수로 확정하지 않는다.

기본 기준은 다음과 같다.

- `id`는 기본 키이므로 인덱스가 생성된다.
- 목록 조회 정렬이 필요하면 `created_at` 인덱스를 검토할 수 있다.
- 검색 기능이 도입되면 `title`, `content` 관련 인덱스를 검토할 수 있다.

현재 단계에서는 게시글 CRUD 검증이 우선이므로  
추가 인덱스는 후속 후보로 둔다.

---

## 12. 삭제 방식

현재 단계에서는 논리 삭제와 물리 삭제 중 하나를 최종 확정하지 않는다.

### 12.1 물리 삭제

물리 삭제는 게시글 삭제 시 DB 행을 실제로 삭제하는 방식이다.

장점은 다음과 같다.

- 구조가 단순하다.
- 현재 CRUD 검증에 적합하다.
- 별도 삭제 상태 컬럼이 필요 없다.

단점은 다음과 같다.

- 삭제 이력 보존이 어렵다.
- 복구 기능을 만들기 어렵다.

### 12.2 논리 삭제

논리 삭제는 게시글 삭제 시 삭제 상태만 변경하는 방식이다.

추가 컬럼 후보는 다음과 같다.

| 컬럼명 | 타입 후보 | 설명 |
|---|---|---|
| deleted | BOOLEAN | 삭제 여부 |
| deleted_at | TIMESTAMP | 삭제 시각 |

장점은 다음과 같다.

- 삭제 이력 보존이 가능하다.
- 복구 기능을 추가하기 쉽다.

단점은 다음과 같다.

- 조회 조건에 삭제 여부 필터가 필요하다.
- 현재 최소 CRUD 구조가 조금 복잡해진다.

### 12.3 현재 결정

현재 단계에서는 단순 CRUD 검증을 우선하므로 물리 삭제를 우선 검토할 수 있다.

다만 논리 삭제 필요성이 확인되면 후속 단계에서 `deleted`, `deleted_at` 컬럼 추가를 검토한다.

## 13. JPA Entity 매핑 방향

현재 단계에서 게시글 Entity는 `PostEntity` 이름을 후보로 둔다.

예상 위치는 다음과 같다.

    post.entity.PostEntity

예상 필드는 다음과 같다.

| Entity 필드 | DB 컬럼 | 타입 후보 |
|---|---|---|
| id | id | Long |
| title | title | String |
| content | content | String |
| createdAt | created_at | LocalDateTime |
| updatedAt | updated_at | LocalDateTime |

현재 단계에서는 Entity와 Domain 객체를 별도로 분리하지 않을 수 있다.  
도메인 규칙이 복잡해지면 분리 여부를 다시 검토한다.

---

## 14. Repository 매핑 방향

현재 단계에서 Repository는 Spring Data JPA 사용을 우선 검토한다.

예상 위치는 다음과 같다.

    post.repository.PostRepository

예상 방향은 다음과 같다.

    PostRepository extends JpaRepository<PostEntity, Long>

현재 단계에서 필요한 기본 기능은 다음과 같다.

- 전체 게시글 목록 조회
- 게시글 단건 조회
- 게시글 저장
- 게시글 삭제
- 게시글 존재 여부 확인

별도 커스텀 Repository는 현재 단계에서 확정하지 않는다.  
검색, 복잡한 조건 조회, 페이징 고도화가 필요해지면 후속으로 검토한다.

## 15. 생성/수정 시각 관리 기준

`created_at`, `updated_at`은 게시글의 기본 관리 정보다.

현재 검토 가능한 방식은 다음과 같다.

### 15.1 JPA Auditing 사용

JPA Auditing을 사용하면 생성 시각과 수정 시각을 자동으로 관리할 수 있다.

검토 기준은 다음과 같다.

- `@CreatedDate`를 사용해 생성 시각을 관리할 수 있다.
- `@LastModifiedDate`를 사용해 수정 시각을 관리할 수 있다.
- 별도 auditing 설정이 필요하다.
- Spring Boot + JPA 구조에서 자연스럽게 사용할 수 있다.

### 15.2 애플리케이션 코드에서 직접 설정

Service 또는 Entity 생성 메서드에서 시각을 직접 설정할 수 있다.

검토 기준은 다음과 같다.

- 구조가 단순하다.
- 현재 최소 CRUD 검증 단계에 적용하기 쉽다.
- 시각 설정 책임이 코드에 명시된다.
- 반복 코드가 생길 수 있다.

### 15.3 현재 결정

현재 문서에서는 최종 방식을 확정하지 않는다.

구현 계획 문서에서 다음 중 하나를 선택한다.

- JPA Auditing 사용
- 애플리케이션 코드에서 직접 설정

---

## 16. Flyway 적용 기준

현재 overview에서 DB 마이그레이션 도구로 Flyway 사용 가능성을 고려하고 있다.

현재 단계의 Flyway 기준은 다음과 같다.

- Flyway 사용 가능성을 열어둔다.
- Flyway를 사용하면 migration SQL은 `src/main/resources/db/migration` 아래에 둔다.
- 최초 게시글 테이블 생성 migration을 작성할 수 있다.
- Flyway 즉시 적용 여부는 구현 계획에서 결정한다.

예상 migration 파일명 후보는 다음과 같다.

    V1__create_posts_table.sql

현재 단계에서는 Flyway 적용을 필수로 확정하지 않는다.  
다만 DB 구조가 문서화되었으므로 Flyway 적용은 자연스러운 후속 선택지가 될 수 있다.

---

## 17. Migration SQL 후보

Flyway를 사용할 경우 최초 migration SQL 후보는 다음과 같다.

    CREATE TABLE posts (
        id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
        title VARCHAR(255) NOT NULL,
        content TEXT NOT NULL,
        created_at TIMESTAMP NOT NULL,
        updated_at TIMESTAMP NOT NULL
    );

위 SQL은 후보이며, 실제 DB 제품에 따라 조정할 수 있다.

예를 들어 MySQL, PostgreSQL, H2에서 ID 자동 증가 구문이나 TEXT 타입 처리 방식이 달라질 수 있다.

---

## 18. DB 제품 결정 기준

현재 단계에서는 특정 운영 DB를 최종 확정하지 않는다.

검토 가능한 DB는 다음과 같다.

- H2
- PostgreSQL
- MySQL 또는 MariaDB

현재 기준은 다음과 같다.

- 개발 초기에는 H2를 사용할 수 있다.
- 실제 운영 또는 재사용 가능한 구조 검증에는 PostgreSQL 또는 MySQL 계열을 검토할 수 있다.
- Flyway를 사용한다면 DB 제품별 SQL 차이를 고려해야 한다.

DB 제품은 구현 계획 또는 환경 설정 문서에서 결정한다.

---

## 19. API와 DB 매핑 기준

API DTO와 DB Entity는 다음 기준으로 연결한다.

### 19.1 등록 요청

`PostCreateRequest`는 다음 DB 컬럼과 연결된다.

| 요청 필드 | DB 컬럼 |
|---|---|
| title | title |
| content | content |

등록 시 `id`, `created_at`, `updated_at`은 시스템에서 생성 또는 설정한다.

### 19.2 수정 요청

`PostUpdateRequest`는 다음 DB 컬럼과 연결된다.

| 요청 필드 | DB 컬럼 |
|---|---|
| title | title |
| content | content |

수정 시 `updated_at`은 갱신되어야 한다.  
`created_at`은 변경하지 않는 것을 원칙으로 한다.

### 19.3 목록 응답

`PostListResponse`는 다음 DB 컬럼과 연결된다.

| 응답 필드 | DB 컬럼 |
|---|---|
| id | id |
| title | title |
| createdAt | created_at |
| updatedAt | updated_at |

목록 응답에는 현재 단계에서 `content`를 포함하지 않는다.

### 19.4 상세 응답

`PostDetailResponse`는 다음 DB 컬럼과 연결된다.

| 응답 필드 | DB 컬럼 |
|---|---|
| id | id |
| title | title |
| content | content |
| createdAt | created_at |
| updatedAt | updated_at |

## 20. 도메인 모델과 DB 매핑 기준

도메인 모델 문서에서 정의한 게시글 속성은 DB 컬럼과 다음처럼 대응된다.

| 도메인 개념 | DB 컬럼 | 설명 |
|---|---|---|
| 게시글 식별자 | id | 게시글 고유 식별자 |
| 게시글 제목 | title | 필수 값 |
| 게시글 본문 | content | 필수 값 |
| 생성 시각 | created_at | 등록 시 기록 |
| 수정 시각 | updated_at | 등록 및 수정 시 기록 |

현재 단계에서 작성자, 상태, 조회수, 삭제 여부는 DB 컬럼에 포함하지 않는다.

## 21. 테스트 데이터 기준

현재 단계에서는 최소 CRUD 검증을 위한 테스트 데이터만 고려한다.

테스트 데이터 예시는 다음과 같다.

| id | title | content |
|---:|---|---|
| 1 | 첫 번째 게시글 | 첫 번째 게시글 본문입니다. |
| 2 | 두 번째 게시글 | 두 번째 게시글 본문입니다. |

테스트 데이터는 다음 용도로 사용할 수 있다.

- 목록 조회 검증
- 상세 조회 검증
- 수정 검증
- 삭제 검증
- 존재하지 않는 게시글 처리 검증

테스트 데이터 삽입 방식은 구현 계획 문서에서 정한다.

## 22. DB 결정 보류 사항

현재 DB 설계에서는 다음 판단을 보류한다.

- 운영 DB 제품 최종 선택
- H2 사용 여부
- PostgreSQL 사용 여부
- MySQL 또는 MariaDB 사용 여부
- Flyway 즉시 적용 여부
- ID 생성 전략 최종 선택
- 생성/수정 시각 자동 관리 방식
- 논리 삭제와 물리 삭제 중 최종 선택
- 삭제 상태 컬럼 도입 여부
- 제목 최대 길이 최종 확정
- 본문 타입 최종 확정
- 공백 문자열 DB 제약 처리 여부
- 추가 인덱스 도입 여부
- 작성자 컬럼 도입 여부
- 조회수 컬럼 도입 여부
- 게시글 상태 컬럼 도입 여부
- DB audit 컬럼 확장 여부

위 항목은 현재 단계에서 성급히 확정하지 않는다.  
필요성이 검증되면 후속 문서에서 구체화한다.
