# docs/projects/board/01-overview.md

## 1. 문서 목적

이 문서는 board 프로젝트의 **가장 기초적인 개요 문서**다.

목적은 다음과 같다.

- board 프로젝트를 한 문서로 빠르게 이해할 수 있게 한다.
- 현재 단계에서 board를 왜 시작하는지 고정한다.
- board의 현재 성격과 검증 방향을 짧게 정리한다.
- 현재 범위와 비포함 범위를 개략적으로 구분한다.
- 이후 요구사항 / 설계 / 구현 계획 문서의 출발점을 제공한다.

이 문서는 상세 요구사항 명세서가 아니다.  
현재는 board 프로젝트의 목적, 성격, 범위, 다음 문서로 이어지는 기준점을 짧게 고정하는 데 집중한다.

---

## 2. board 한 줄 요약

board는 specflow 방식이 실제 애플리케이션 개발에도 유효한지 검증하기 위한  
**첫 번째 실제 애플리케이션 프로젝트**다.

---

## 3. 왜 board를 시작하는가

specflow는 문서 기반 AI 협업 흐름을 다루는 도구다.  
하지만 이 방식이 실제 프로젝트에서도 작동하는지는  
별도의 실제 대상이 있어야 검증할 수 있다.

board를 시작하는 이유는 다음과 같다.

- 문서 우선 개발 흐름을 실제 프로젝트에 적용해 본다.
- 요구사항 / 설계 / 구현 계획 / 상태 / 히스토리 구조를 실제로 운영해 본다.
- AI가 실제 프로젝트 문서를 기준으로 협업할 때 필요한 최소 문서 구조를 확인한다.
- 이후 다른 프로젝트에도 재사용 가능한 운영 방식을 검증한다.

즉 board는 단순 예시가 아니라,  
**specflow 검증을 위한 실제 프로젝트 작업공간**이다.

---

## 4. 현재 board의 성격

현재 board는 다음 성격으로 이해한다.

- 실제 애플리케이션 프로젝트 문서군
- specflow 적용 검증 대상
- 최소 구현 범위를 기준으로 진행하는 알파 단계 프로젝트
- Spring Boot 기반 레이어드 구조를 우선 검증하는 프로젝트

현재 단계에서 board를 완성된 제품처럼 다루지는 않는다.  
우선은 **작동 가능한 최소 구조와 문서 흐름이 실제 프로젝트에서도 유지되는지 검증하는 프로젝트**로 본다.

또한 board는 specflow의 하위 기능이 아니다.  
board는 specflow가 다루는 **독립적인 실제 애플리케이션 프로젝트 문서군**으로 이해해야 한다.

---

## 5. 현재 범위

현재 `01-overview.md` 에서 우선 다루는 범위는 다음과 같다.

- board 프로젝트의 목적과 현재 성격 정리
- board 문서군의 최소 범위와 경계 정리
- 후속 요구사항 / 설계 / 구현 계획 문서로 이어지는 출발점 마련
- 공통 standards 문서와의 연결 기준 정리

현재 범위에 아직 직접 포함하지 않는 것은 다음과 같다.

- 상세 기능 요구사항 확정
- 상세 화면 설계 확정
- 상세 API 명세 확정
- 상세 DB 스키마 확정
- 실제 구현 코드 본격 착수
- 운영/배포 구조 설계
- 장기 확장 구조 확정

즉 현재는 상세 구현안을 확정하기보다,  
**board 문서군이 어떤 순서와 경계로 확장될지 잡는 단계**에 가깝다.

---

## 6. board와 다른 문서의 관계

### 6.1 `README.md`
- board 전체 정체성과 방향을 설명하는 진입 문서

### 6.2 `AGENTS.md`
- board 문서를 다룰 때의 전용 작업 규칙 문서

### 6.3 `index.md`
- board 문서 목록과 읽기 순서를 안내하는 인덱스 문서

### 6.4 `01-overview.md`
- board 프로젝트를 가장 짧게 이해하기 위한 출발 문서
- 이후 요구사항 / 설계 / 구현 계획 문서의 기준점

### 6.5 `docs/projects/standards/**`
- 여러 프로젝트가 공통으로 참고하는 개발 기준 문서
- board 문서는 standards 본문을 복사하기보다, board에 어떻게 적용할지를 정리한다

이 관계에서 중요한 점은 다음과 같다.

- `01-overview.md` 는 board의 기초 개요를 다루는 문서다.
- 상세 요구사항, 구조 설계, 구현 순서는 각각 후속 문서에서 분리해 다룬다.
- specflow 자체 운영 규칙은 board 문서가 아니라 상위 공통 문서와 specflow 문서에서 다룬다.

---

## 7. 다음 문서로 이어지는 방향

이 문서 다음에는 보통 아래 방향으로 이어진다.

1. 범위와 목적을 더 구체화하는 요구사항 문서
2. 구조와 책임을 정리하는 설계 문서
3. 실제 작업 순서를 나누는 구현 계획 문서
4. 진행 상황과 판단을 기록하는 상태 / 히스토리 문서

이때 중요한 원칙은 다음과 같다.

- 요구사항 문서가 설계 문서를 대신하지 않는다.
- 설계 문서가 구현 계획 문서를 대신하지 않는다.
- 상태 문서는 현재 진행 상황을 기록하고, 개요 문서는 프로젝트 성격과 범위를 고정한다.
- standards 문서는 공통 기준을 제공하고, board 문서는 그 적용 판단을 기록한다.

즉 `01-overview.md` 는  
board 문서 구조의 가장 앞쪽에서  
**“이 프로젝트를 왜 시작하는가”와 “현재 어디까지를 다루는가”를 고정하는 문서**다.

---

## 8. 최종 정리

board는 specflow의 하위 기능이 아니라  
**specflow가 다루는 첫 번째 실제 애플리케이션 프로젝트**다.

이 문서는 그 프로젝트를 시작할 때 가장 먼저 공유해야 하는  
기초 개요를 정리하는 문서다.

현재 단계에서 이 문서의 역할은 상세 설계를 담는 것이 아니라,  
board의 정체성, 현재 범위, 비포함 범위, 후속 문서 연결 방향을  
짧고 분명하게 고정하는 데 있다.
