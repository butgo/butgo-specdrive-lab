# SpecDrive v2.0 아키텍처 개발 방법론 대안 검토

## 1. 문서 목적
본 문서는 SpecDrive v2.0의 3대 핵심 과제(프롬프트 정규화, 기획 파서 확장, 위저드 UI)를 구현하기 위한 최적의 접근 방식을 탐색하고, 아키텍트의 의사결정을 돕기 위한 대안을 제시한다.

---

## 2. 방법론 대안 비교 (Alternatives)

### 대안 1: 화면 우선 역산 설계 (UI-First / Mockup-Driven)
**순서:** 위저드 공통 UI (더미 데이터) ➡️ 프롬프트 정규화 ➡️ 파서 확장

*   **개념:** 데이터를 먼저 상상하기보다, 사용자가 실제로 클릭하고 입력할 '화면(Webview)'을 먼저 그려놓고 필요한 데이터를 역추적하는 방식.
*   **장점:**
    *   백엔드(파서/스키마)를 과도하게 복잡하게 설계하는 오버엔지니어링(Over-engineering)을 방지.
    *   눈에 보이는 폼(Form)을 통해 직관적이고 빠른 피드백 확보 ("이 입력칸은 불필요하다", "체크박스가 낫겠다" 등).
*   **단점:** 화면 변경에 따라 백엔드 스키마가 계속 흔들릴 수 있음.

### 대안 2: 수직적 기능 분할 (Vertical Slicing / Agile)
**순서:** [항목 1개 UI + 정규화 + 파서] ➡️ [항목 2개 UI + 정규화 + 파서]

*   **개념:** 시스템을 수평(데이터, 로직, UI 계층)으로 구축하는 대신, 아주 작은 기능 단위 하나를 수직으로(End-to-End) 먼저 관통시키는 애자일 방식.
*   **장점:**
    *   전체 파서 완성을 기다릴 필요 없이, 핵심 기능 하나가 동작하는 것을 가장 빠르게 확인 가능.
    *   파서와 UI 간의 브릿지 통신 문제를 조기 발견 및 해결.
*   **적용 예시:** 전체 기획서 파싱 대신, "프로젝트 기술 스택" 단일 항목의 UI 입력부터 파싱까지 먼저 1사이클 완성.

### 대안 3: 스키마 주도 동적 UI (JSON-Schema Driven UI) - ★가장 추천
**순서:** JSON Schema 정의 ➡️ UI 자동 생성 & 파서 연동

*   **개념:** 위저드에서 입력받을 데이터 규격을 순수한 **`JSON Schema`** 파일로만 정의하고, 프론트엔드가 이를 해석하여 자동으로 화면을 렌더링하는 방식. Spec-Driven Development 철학의 궁극적 형태.
*   **동작 방식:**
    *   Webview 개발 시 폼을 수동으로 그리지 않음.
    *   `react-jsonschema-form` 등 라이브러리 활용.
*   **장점:**
    *   향후 위저드 입력 항목이 추가/변경되더라도 UI 코드 수정 불필요.
    *   스키마(JSON) 수정만으로 화면과 파서가 동시에 적응하는 궁극의 유연성 확보.

---

## 3. 결론 및 권장 사항

초기 안정성을 중시한다면 전통적인 Waterfall 방식을, 빠른 시각적 피드백과 심리적 진척도를 원한다면 **대안 1 (UI-First)**을 권장한다. 

그러나 SpecDrive의 궁극적인 비전("문서가 코드를 주도한다")을 고려할 때, 시스템 아키텍처 관점에서 가장 우아하고 확장성이 뛰어난 **대안 3 (JSON-Schema Driven UI)** 방식을 최종 목표 모델로 채택하는 것을 강력히 권장한다.
