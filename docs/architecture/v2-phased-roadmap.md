# SpecDrive v2.0 아키텍처 구축 로드맵 (단계별 우선순위)

## 1. 문서 목적
본 문서는 SpecDrive(VSCode 플러그인)의 기본 PoC(개념 증명) 완료 이후, 시스템을 고도화하기 위한 3가지 핵심 과제(프롬프트 정규화, 기획 파서 확장, 위저드 UI)의 아키텍처적 선후 관계와 단계별 실행 목표를 정의한다.

---

## 2. 아키텍처 구축 원칙: "데이터 흐름의 상류(Upstream) 통제"
시스템의 확장성과 안정성을 위해, 데이터가 정의되는 가장 앞단(Schema)부터 설계하고 이를 기반으로 파서(Logic)와 화면(UI)을 순차적으로 구축한다.

---

## 3. 단계별 실행 계획 (Phase Roadmap)

### 🥇 [Phase 1] 위저드(Wizard) 프롬프트 정규화 및 공통 데이터 스키마 설계
**모든 후속 작업의 뼈대가 되는 가장 중요한 1순위 과제.** AI에게 전달할 데이터의 포맷과 규격을 확정한다.

*   **목표:** 기획(Doc) 단계와 개발(Dev) 단계에서 각각 어떤 입력값이 필요한지 정의하고, 이를 JSON 또는 정형화된 마크다운 템플릿 구조로 설계한다.
*   **주요 작업 (To-Do):**
    *   **Doc(기획) 모드 필수 입력값 정의:**
        (예: 프로젝트 목적, 타겟 사용자, 핵심 기술 스택, 성능 제약사항 등)
    *   **Dev(개발) 모드 필수 입력값 정의:**
        (예: 수정 대상 파일명, 참조해야 할 API 문서 경로, 테스트 시나리오 등)
    *   위 항목들을 바탕으로 공통 데이터 스키마(Schema) 문서 작성.

### 🥈 [Phase 2] Doc(기획) 파서 확장 (Advanced Spec-Driven Parser)
Phase 1에서 정의한 규격화된 문서(Markdown)를 프로그래밍 객체(Object)로 완벽하게 변환하는 로직을 구축한다.

*   **목표:** 단순한 Task 추출(`markdownParser.ts`)을 넘어, 마크다운 문서 전체의 구조(Phase/Cycle/Task)를 추상 구문 트리(AST) 수준으로 이해하는 파서 개발.
*   **주요 작업 (To-Do):**
    *   마크다운 헤더(`##`, `###`) 구조를 기반으로 부모-자식 관계를 인식하는 파싱 로직 구현.
    *   "이 Task는 프론트엔드 작업이고, 저 Task는 DB 작업이다"를 구분하여 상태 제어기(`stateMachine.ts`)가 알맞은 프롬프트를 선택할 수 있도록 컨텍스트(Context) 제공 기능 강화.

### 🥉 [Phase 3] 위저드 공통 UI (Webview) 구축
Phase 1과 Phase 2의 기반 위에, 개발자가 편리하게 데이터를 입력할 수 있는 프론트엔드 인터페이스를 개발한다.

*   **목표:** VSCode Webview API를 활용하여, 정해진 스키마(Phase 1)에 맞춰 직관적으로 입력할 수 있는 다이나믹 폼(Dynamic Form) UI 제공.
*   **주요 작업 (To-Do):**
    *   React, Vue 또는 순수 HTML/JS를 이용한 Webview 패널 제작.
    *   (예시) 사용자가 "Doc 모드" 버튼을 누르면 Phase 1에서 정의한 5가지 입력 폼이 화면에 표시.
    *   사용자 입력을 JSON으로 직렬화(Serialize)하여 플러그인 백엔드(상태 제어기)로 안전하게 전송하는 브릿지(Bridge) 통신 로직 구현.
